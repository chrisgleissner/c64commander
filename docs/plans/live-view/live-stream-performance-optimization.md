# Implement and Prove a Low-CPU, Loss-Resilient, Bounded-Latency Ultimate 64 Streaming Pipeline

You are an expert in real-time audio/video systems, UDP networking, Android, iOS, Capacitor, TypeScript, Kotlin, Swift, Rust, native audio APIs, GPU rendering, performance profiling, concurrency, deterministic testing, CI/CD infrastructure, hardware-in-the-loop testing, telemetry, and statistical performance analysis.

Your task is to inspect, instrument, profile, redesign where evidence justifies it, implement, test, benchmark, document, commit, push, and prepare a merge-ready pull request for the Ultimate 64 audio/video streaming feature in this repository.

Do not treat any proposed architecture as automatically correct. Begin by measuring the existing implementation.

Do not claim completion based on code inspection, passing unit tests, or one successful manual run. Completion requires automated, reproducible evidence from deterministic tests, controlled performance tests, and a mandatory Pixel 4-to-Ultimate 64 hardware-in-the-loop test.

The resulting test system must continue running on subsequent builds so that correctness, latency, CPU consumption, concealment behaviour, and non-accumulating playback delay cannot silently regress.

---

# 1. Non-negotiable outcomes

The completed system must satisfy all of the following.

## 1.1 Audio continuity

Audio output must be continuous.

Under the declared supported UDP loss, jitter, reordering, and CPU-load profiles:

* There must be no audio callback underruns.
* There must be no application-induced audio loss.
* There must be no clicks caused by buffer boundaries, stale data, concealment transitions, or incorrect clock correction.
* There must be no playback of data from an earlier stream session.
* Every required output sample position must contain either:

  * a valid received sample, or
  * a deliberately generated concealment sample.

Missing source audio must never create an unintended gap in the playback timeline.

## 1.2 Video continuity

At the selected effective presentation rate, every scheduled presentation slot must contain an output frame.

An output frame may be:

* fully reconstructed from received source packets,
* partially reconstructed with concealed missing regions,
* a deliberately concealed replacement frame,
* a deliberately repeated previous frame,
* or a frame generated by another evidence-backed concealment method.

Do not claim that an exact source frame was rendered when packets required to reconstruct it were irretrievably lost.

The enforceable requirement is continuous presentation cadence, not impossible recovery of unavailable source information.

## 1.3 Stable bounded latency

There must never be an ever-growing delay between the Ultimate 64 sender and device playback.

For both audio and video:

```text
e2e_latency_p99 < 30 ms
```

This must hold within the explicitly declared supported operating envelope.

Latency must remain bounded during:

* clean LAN operation,
* packet loss,
* packet reordering,
* packet jitter,
* temporary renderer slowdown,
* temporary CPU pressure,
* thermal pressure,
* UI activity,
* stream restart,
* PAL/NTSC changes,
* and source switching.

The system must prefer concealment or controlled video degradation over accumulating playback delay.

No queue may grow indefinitely.

No component may attempt to preserve old data by allowing playback to drift progressively further behind the sender.

## 1.4 Low CPU consumption and responsiveness

The system must minimise CPU use while preserving audio continuity and the highest safe video cadence.

The weakest supported devices must remain responsive.

The optimisation must reduce or avoid:

* unnecessary copies,
* per-packet allocations,
* per-frame allocations,
* JSON media transfer,
* Base64 media transfer,
* high-frequency Capacitor calls,
* high-frequency JNI or Swift FFI calls,
* redundant pixel conversion,
* conversion of frames that will not be displayed,
* excessive thread dispatch,
* excessive timer wakeups,
* logging in hot paths,
* busy waiting,
* unnecessary locks,
* unbounded queues,
* repeated texture allocation,
* repeated audio-buffer allocation,
* and avoidable garbage collection.

## 1.5 Automated regression prevention

Every eligible build must run automated tests that prove:

* protocol correctness,
* deterministic concealment,
* continuous output cadence,
* audio continuity,
* bounded queue behaviour,
* stable latency,
* low CPU use,
* adaptive frame-rate behaviour,
* telemetry correctness,
* and user-visible Stats correctness.

A mandatory HIL stage must test a physical Pixel 4 receiving a live stream from a physical Ultimate 64.

A build must not be reported as fully green if the mandatory HIL test was skipped, unavailable, or produced invalid measurements.

Infrastructure failure may be reported separately from product failure, but it must not be converted into a passing result.

# 2. Required interpretation of packet loss

UDP cannot guarantee packet delivery.

Do not promise exact reconstruction of information that never arrived.

Instead, distinguish these cases explicitly:

1. Source packet loss before the application receives the packet.
2. Operating-system or socket receive-buffer loss.
3. Application-induced loss.
4. Packet rejected as malformed.
5. Packet rejected as too late.
6. Duplicate packet.
7. Reordered packet successfully recovered.
8. Audio concealment.
9. Video region concealment.
10. Whole-frame concealment.
11. Intentional frame-rate decimation.
12. Renderer-backlog replacement.
13. Lifecycle or source-switch cancellation.

The implementation must guarantee:

* zero avoidable application-induced loss within the supported load envelope,
* deterministic concealment for genuinely missing source content,
* continuous audio output,
* continuous video presentation slots at the effective presentation cadence,
* bounded latency,
* and complete telemetry describing what actually occurred.

Never combine concealment, intentional decimation, and defective drops into one generic “dropped frames” counter.

# 5. Measurement integrity

## 5.1 True end-to-end latency

Do not label any of the following as true end-to-end latency:

* packet receive to render callback,
* frame completion to texture upload,
* native callback to JavaScript,
* application enqueue to audio callback,
* or decoder completion to presentation.

True end-to-end latency must begin at the Ultimate 64 source timeline and end at actual device presentation.

The implementation must establish a defensible measurement method for both audio and video.

Acceptable approaches include:

1. Source timestamps embedded in packets and mapped to the device monotonic clock.
2. A test-only Ultimate 64 sender timestamp or sequence marker plus a validated clock-mapping protocol.
3. A common-clock external HIL fixture.
4. Another method that is demonstrated to measure source-to-presentation latency with a quantified error bound.

The clock relationship must account for:

* clock offset,
* clock drift,
* timestamp wraparound,
* network asymmetry,
* and calibration uncertainty.

The measurement method must report its estimated uncertainty.

The automated latency gate is invalid if the uncertainty is too large to distinguish compliance with the 30 ms threshold.

As a starting requirement, measurement uncertainty should be below 2 ms. If this is not achievable, document the limitation and do not claim that the 30 ms requirement has been proven.

## 5.2 Audio presentation time

Where platform APIs provide playback-head or hardware timestamps, use them rather than assuming that writing an audio buffer means the samples were played.

Account for:

* queued audio,
* device hardware buffering,
* sample-rate conversion,
* audio callback timing,
* and known platform output latency.

## 5.3 Video presentation time

Measure actual or best-available display presentation time rather than texture-upload completion.

Use presentation timestamps, frame fences, compositor timing, display callbacks, or another platform-supported mechanism.

Document exactly what the final timestamp represents.

## 5.4 Monotonic clocks

All durations and latency calculations must use monotonic clocks.

Wall-clock time may be recorded for human-readable logs but must not drive correctness or performance calculations.

# 6. Latency budget

Create an explicit latency budget covering:

```text
source packetisation
+ network transit
+ socket receive delay
+ packet reordering
+ assembly
+ concealment decision
+ audio/video queue residence
+ conversion
+ renderer/audio-device scheduling
+ hardware presentation
```

The combined p99 must remain below 30 ms.

Do not hide a large jitter buffer inside the implementation.

Every queue must have:

* a documented maximum item count,
* a documented maximum age in milliseconds,
* a defined overflow policy,
* telemetry for current and maximum residence time,
* and an invariant preventing old data from being presented after it has exceeded its deadline.

If a packet, audio block, or frame cannot be presented inside the latency budget:

* do not allow it to extend future latency,
* discard or replace it according to the defined policy,
* conceal the affected output timeline,
* and record the event.

The implementation must never “catch up later” by retaining an ever-growing backlog.

# 7. No-drift and no-growing-lag invariants

Implement and test all of the following invariants:

1. Queue capacities are bounded.
2. Queue age is bounded independently of queue item count.
3. Audio playback is driven by a stable output clock.
4. Source and output clock drift is corrected without increasing queue delay.
5. Video is scheduled against the selected master clock.
6. Late video is replaced with the newest deadline-eligible frame or a concealed frame.
7. Late audio input is concealed rather than inserted after its playback deadline.
8. Reordering is attempted only within a strictly bounded time window.
9. Jitter-buffer adaptation may move within a defined bounded range but may not grow without limit.
10. Restarting or switching source clears all stale session state.
11. The governor may reduce video cost but may not increase audio delay to protect video.
12. No shutdown or background transition leaves stale media queued for later playback.

Long-running tests must demonstrate that latency does not trend upward.

For each soak test:

* compute rolling p50, p95, and p99 latency,
* compute a linear latency-drift slope,
* compare the beginning and end of the run,
* and fail if there is evidence of progressive lag.

At minimum:

* every rolling one-minute latency window must satisfy `p99 < 30 ms`,
* the final five-minute p99 must not exceed the first five-minute p99 by more than 2 ms,
* and the measured positive latency trend must remain within the committed tolerance.

The final tolerance must be derived from measurement noise and committed as an explicit test threshold. It must not be silently widened to pass a regression.

* short-term waveform extrapolation,
* repetition with cross-fading,
* ramp-to-silence and ramp-from-silence,
* frequency-domain or predictor-based concealment,

Select the method using objective evidence.

Evaluate:

* click and discontinuity magnitude,
* spectral discontinuity,
* repeated-waveform artefacts,

The concealment algorithm must:

* use bounded state,
* produce exactly the required output sample count,
* never reuse uninitialised or stale memory,
* distinguish concealed samples from received samples,
* and transition cleanly back to valid audio.

Do not add retransmission waits that threaten the latency budget.

# 11. Adaptive performance governor

Implement a hybrid adaptive governor.

## 11.1 User modes

Expose:

```text
Video frame rate
- Auto, up to 100% (recommended)
- 100%
- 50%
- 25%
```

Use the actual detected source cadence.

For example:

| Source |               100% |              50% |                 25% |
| ------ | -----------------: | ---------------: | ------------------: |
| PAL    |             50 fps |           25 fps |            12.5 fps |
| NTSC   | actual source rate | half source rate | quarter source rate |

Do not internally replace 59.94 Hz with 60 Hz where doing so creates drift.

Use deterministic cadence division:

* 100% presents each source slot.
* 50% presents every second source slot.
* 25% presents every fourth source slot.

## 11.2 Manual mode semantics

A manually selected rate is a requested maximum.

It is not permission to:

* lose audio,
* exceed the latency budget,
* exhaust CPU,
* build a backlog,
* or freeze the UI.

If required to protect audio or bounded latency, the governor may temporarily reduce effective video work below the requested maximum.

The system must expose:

* requested mode,
* effective mode,
* reason for override,
* time of transition,
* and recovery status.

## 11.3 Auto mode

Auto must attempt full source cadence whenever it is safe.

It must consider:

* audio-ring fill and trend,
* audio callback timing,
* near-underrun events,
* receive-thread scheduling delay,
* packet queue pressure,
* frame-assembly pressure,
* frame processing time,
* renderer queue pressure,
* queue residence time,
* process CPU time,
* per-thread CPU time,
* UI jank,
* thermal state,
* allocation rate,
* garbage-collection pauses,
* and measured end-to-end latency.

Do not base adaptation on one instantaneous CPU sample.

## 11.4 Adaptation order

Use controlled degradation such as:

```text
100% -> 50% -> 25%
```

Audio remains full-rate.

UDP receive and sequence accounting remain complete.

Do not stop parsing packets merely because fewer video frames are presented if doing so would damage:

* audio extraction,
* packet-loss accounting,
* source timing,
* concealment decisions,
* or recovery.

## 11.5 Hysteresis

Demotion must be faster than promotion.

Demote when:

* an audio underrun occurs,
* audio fill crosses a critical threshold,
* latency approaches the hard budget,
* processing consistently misses deadlines,
* or queue age approaches its maximum.

Promote only after sustained evidence of headroom.

Use:

* minimum stable windows,
* transition cooldowns,
* one-level-at-a-time promotion,
* and explicit anti-oscillation tests.

All policy thresholds must be documented and testable.

## 11.6 Calibration

Runtime adaptation is mandatory.

Startup-only calibration is insufficient because:

* thermal conditions change,
* background load changes,
* operating-system scheduling changes,
* battery and charging conditions change,
* and AppSupport overhead may vary.

A previously stable result may be cached as a starting hint.

The cache key must include relevant characteristics such as:

* device model,
* operating-system version,
* application version,
* renderer implementation,
* source mode,
* source format,
* and performance-affecting settings.

Runtime telemetry remains authoritative.

# 16. Performance regression gates

Performance thresholds must be committed to version control in a machine-readable configuration.

Each threshold must identify:

* test profile,
* device or runner,
* metric,
* unit,
* required maximum or minimum,
* measurement duration,
* warm-up period,
* number of repetitions,
* and statistical aggregation.

Do not use one noisy run as a hard performance conclusion.

Use repeated runs and robust statistics.

Where appropriate:

* perform warm-up,
* run multiple repetitions,
* use median or trimmed mean for stable aggregate CPU measurements,
* retain p95 and p99 latency,
* and report confidence or observed variance.

## 16.1 Mandatory latency gates

For every mandatory supported profile:

```text
audio_e2e_latency_p99 < 30 ms
video_e2e_latency_p99 < 30 ms
```

Also require:

* no rolling-window breach,
* no progressive latency growth,
* bounded queue residence,
* and valid measurement uncertainty.

## 16.2 Mandatory audio gates

Require:

```text
audio_callback_underruns = 0
application_induced_audio_loss = 0
unexplained_audio_gaps = 0
stale_session_audio_samples = 0
```

Packet loss may result in non-zero:

```text
concealed_audio_samples
```

That is acceptable only when:

* concealment matches the loss timeline,
* output sample count remains correct,
* continuity tests pass,
* and click/discontinuity thresholds pass.

## 16.3 Mandatory video gates

At the effective presentation cadence:

```text
unexplained_missing_presentation_slots = 0
```

Source packet loss may result in:

* partial concealment,
* whole-frame concealment,
* or deliberate frame repetition.

These must be reported accurately.

Intentional 50% or 25% cadence selection is not a defect.

## 16.4 CPU gates

Establish device-specific absolute CPU ceilings and relative regression ceilings from the measured implementation.

The committed test configuration must include:

* Pixel 4 total process CPU ceiling,
* critical-thread CPU ceilings,
* host benchmark ceilings,
* UI responsiveness thresholds,
* and maximum acceptable regression relative to the accepted baseline.

Do not claim “low CPU” using only a relative improvement if the absolute result still makes the device unresponsive.

Do not claim success using only an absolute ceiling if a substantial regression remains hidden below it.

Use both.

CPU must be expressed unambiguously, for example:

* CPU-core equivalents,
* thread CPU nanoseconds per second,
* or a clearly documented platform metric.

## 16.5 Responsiveness gates

Measure and gate:

* UI input response,
* main-thread stalls,
* missed UI frames,
* application-not-responding events,
* stop latency,
* settings interaction while streaming,
* and Stats display interaction.

Opening Stats must not push audio, latency, or CPU outside the accepted envelope.

## 16.6 Memory gates

Require:

* no unbounded memory growth,
* no unbounded native-buffer growth,
* no queue growth,
* no texture leak,
* no audio-buffer leak,
* and no stale session ownership.

# 19. Fact-driven optimisation workflow

Follow this order.

1. Inspect repository structure and existing tests.
2. Identify all streaming, bridge, rendering, audio, telemetry, and lifecycle code.
3. Run the existing test suite.
4. Establish current behaviour and performance baselines.
5. Add missing low-overhead telemetry.
6. Create deterministic packet replay.
7. Create or validate the HIL fixture.
8. Establish a valid end-to-end latency measurement.
9. Profile the current implementation on Pixel 4 and host.
10. Identify measured bottlenecks.
11. Write a concise design decision record.
12. Implement the smallest change that tests the primary hypothesis.
13. Run correctness, performance, and HIL tests.
14. Retain the change only if evidence supports it.
15. Continue iteratively.
16. Implement deterministic concealment.
17. Implement bounded latency and no-drift enforcement.
18. Implement the adaptive governor.
19. Implement Stats and historical telemetry.
20. Add CI/CD gates.
21. Run clean, impaired, and load profiles.
22. Run extended soak tests.
23. Review the entire diff.
24. Document metric definitions and thresholds.
25. Commit logically separated changes.
26. Push to a new branch.
27. Raise a merge-ready pull request.

Do not combine multiple speculative rewrites into one unmeasurable change.

Do not weaken tests to pass the implementation.

Do not silently change acceptance thresholds.
