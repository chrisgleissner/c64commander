# Android Responsiveness Stabilization Plan

Date: 2026-05-06
Status: Ready for execution
Primary spec: [research.md](./research.md)
Expected change classification: `DOC_PLUS_CODE`, `UI_CHANGE`

## 1. Objective

Implement the full Android responsiveness stabilization defined by [research.md](./research.md) so that every issue in the severity-ordered punch list (Section 6 of the research) is addressed in production code, tests, and documentation, and so that live measurements on a real Pixel 4 against a real `u64` confirm a measurable improvement over the baseline captured in Section 5 of the research.

The shipped end state must include fixes for every research ID, in the order defined in section 4 below:

- R-HTTP-1, R-HTTP-2, R-HTTP-3 — HTTP transport
- R-MOD-1, R-MOD-2, R-MOD-3 — module size and re-render scope
- R-BUN-1, R-BUN-2 — bundle composition
- R-RT-1, R-RT-2, R-RT-3, R-RT-4, R-RT-5, R-RT-6, R-RT-7, R-RT-8 — runtime patterns

## 2. Execution Rules

- [research.md](./research.md) is authoritative for the problem statement, evidence, and acceptance criteria.
- This plan is authoritative for sequencing, gating, and validation cadence.
- Every research ID listed in section 6 of the research must be addressed. None may be silently dropped.
- After **each phase**, run the phase-specific test gate defined in section 5 below. A phase is not complete until its gate is green.
- Do not introduce arbitrary sleeps, weaken assertions, comment-out failing tests, or hide failures.
- Do not silently swallow exceptions; surface failures with context per the CLAUDE.md exception-handling rule.
- Every bug fix or migration edge case discovered during implementation must get a targeted regression test in the narrowest deterministic suite that proves it.
- Before declaring completion, re-run the **full live measurement suite** from section 5 of the research and record before/after numbers in `WORKLOG.md` (or this plan's section 8 results table) for at least:
  - cold-start time (`adb shell am start -W`)
  - 12-stroke slider drag frame stats (`dumpsys gfxinfo`)
  - 5-tab navigation frame stats
  - 16-stroke isolated slider drag frame stats (no Telnet)
- Final validation must include `npm run lint`, `npm run test`, `npm run test:coverage`, `npm run build`, and an APK deploy to the attached Pixel 4 with on-device validation per AGENTS.md Phase 5a.
- Global branch coverage must remain `>= 91%` after every phase, not just at the end.
- Preserve the slider stabilization already in place — do not regress `useDeviceBoundSlider`, the firmware-error-aware write path, or the migrated call sites.
- Stop at any phase boundary if a regression appears that cannot be reasoned about in scope; document the blocker rather than papering over it.

### Phase 1. HTTP transport

Goal:

- remove the per-request native plugin overhead from C64U REST calls and tighten timeout budgets so user-perceived control latency drops

Implementation targets:

- **R-HTTP-1**: set `CapacitorHttp.enabled = false` in [capacitor.config.ts](../../../../capacitor.config.ts) (and verify the regenerated `android/app/src/main/assets/capacitor.config.json`); confirm `c64api` continues to function via direct `fetch()` against the configured `androidScheme: "http"` host
- **R-HTTP-2**: ensure no remaining call site relies on CapacitorCookies; if the `@capacitor/cookies` package is still pulled in transitively, deny-list the C64U host or stub it for LAN URLs
- **R-HTTP-3**: introduce `INTERACTIVE_CONTROL_TIMEOUT_MS = 1500`, `BACKGROUND_REQUEST_TIMEOUT_MS = 3000`, keep `UPLOAD_REQUEST_TIMEOUT_MS = 5000` and `PLAYBACK_REQUEST_TIMEOUT_MS = 5000`. Plumb the new constants through the request paths in [c64api.ts](../../../../src/lib/c64api.ts) and update the discovery probe with progressively shorter retry budgets
- add `tests/unit/capacitorConfig.test.ts` that fails when `CapacitorHttp.enabled` is `true` without a documented exemption comment in `capacitor.config.ts`
- update or extend `tests/unit/c64api.test.ts` to assert the new timeout buckets
- regenerate Capacitor sync (`npm run cap:build`)

Phase-1 gate:

- `npm run lint`
- `npm run test -- src/lib/c64api src/lib/connection tests/unit/capacitorConfig.test.ts tests/unit/c64api.test.ts tests/unit/connection`

### Phase 2. Runtime correctness

Goal:

- eliminate silent failure modes (DNS, ENOENT, console-undefined, swallowed startup errors) and make connection failure actionable to the user

Implementation targets:

- **R-RT-1 mDNS**: add `MdnsResolverPlugin.kt` exposing `resolve(host: String): { ip: string; ttlMs: number }` backed by Android `NsdManager`; add `src/lib/native/mdnsResolver.ts` web stub; in [src/lib/c64api.ts](../../../../src/lib/c64api.ts) and `src/lib/connection/connectionManager.ts`, when the platform is Android and the configured host is a bare name, run `MdnsResolverPlugin.resolve(host)` first; on failure, surface a user-actionable error via `addErrorLog` rather than logging at debug. Update [README.md](../../../../README.md) Android section to recommend IP entry until the local network supports mDNS.
- **R-RT-2 ENOENT**: replace the smoke-mode unconditional `Filesystem.readFile` with `Filesystem.stat` (catch ENOENT silently as expected absence) followed by `readFile` only when stat succeeds. Apply the same pattern to any other optional-file probe.
- **R-RT-3 Console "undefined" spam**: instrument `console.log/info/warn/error/debug` during a Telnet menu walk to capture the exact call site (the existing `Capacitor/Console` plugin captures all five). Replace the offending call with structured `addLog(...)`. Add an ESLint rule banning `console.log` in `src/lib/telnet/**` and `src/lib/diagnostics/**` (allow only via `addLog` or `addErrorLog`).
- **R-RT-7 Transport error normalization**: introduce `normalizeTransportError(error, context): TransportFailure` distinguishing DNS, no-route, refused, reset/EPIPE, timeout. Use the result to populate the OFFLINE banner reason and the diagnostics ring buffer.
- **R-RT-8 MainActivity fail-fast**: in `MainActivity.ensureCapacitorPluginAssetPath`, distinguish recoverable (file-already-correct) from unrecoverable (cannot create directory). On the unrecoverable branch, throw `IllegalStateException` and let `onCreate` propagate. Add an Android JVM unit test under `android/app/src/test/java/.../MainActivityAssetPathTest.kt`.

Phase-2 gate:

- `npm run lint` (new ESLint rule and existing checks)
- `npm run test -- src/lib/connection src/lib/native src/lib/c64api src/lib/smoke tests/unit/connection tests/unit/c64api`
- `cd android && ./gradlew testDebugUnitTest`
- `npm run test:coverage` >= 91% global branch
- on-device probe via Pixel 4: cold-start logcat — zero `Msg: undefined` lines during a 30-second Home page session, zero ENOENT stack traces for `c64u-smoke.json`
- on-device probe via Pixel 4 with hostname `u64` (mDNS path): app reaches REAL_CONNECTED within ≤ 3 s; hostname `c64u` (also mDNS) test optional; if both fail the user sees a clear DNS error not a silent OFFLINE
- `npm run cap:build` succeeds

### Phase 5. Background polling and reconciliation

Goal:

- prevent background work (drives polling, Telnet menu walks) from competing with foreground rendering; make the optimistic-override store's reconciliation tolerant of trim/type drift

Implementation targets:

- **R-RT-4 Telnet off the render thread**: cache `useTelnetActions()` support detection per connection lifetime; consolidate session-startup probes; marshal completed Telnet reads through a `MessageChannel.postMessage` microtask or `requestIdleCallback` so they never land inside a frame budget. Update `src/lib/telnet/telnetSession.ts` and `src/pages/HomePage.tsx`.
- **R-RT-5 Drives polling pause during drag**: add a `pollingPauseRegistry` in `src/lib/query/c64PollingGovernance.ts` that exposes `acquirePause()` returning a release handle. `useDeviceBoundSlider` acquires a pause on first drag tick and releases on commit + reconciliation settle. While paused, drives polling and any other interval-driven `refetchInterval` skip their tick. Idle interval may rise from 30 s to 60 s when no recent user mount/unmount.
- **R-RT-6 trim/coerce-aware compare**: in [src/hooks/useAuthoritativeConfigValueState.ts](../../../../src/hooks/useAuthoritativeConfigValueState.ts), replace the strict `Object.is` clear with a per-item equality function defaulting to `(a, b) => normalize(a) === normalize(b)` where `normalize` trims strings and coerces single-token numerics. Allow callers to supply a custom equality function for non-default cases. Audit every reader of `authoritativeValues.pending`; either migrate to a self-resolving mutation flag or accept the new equality function.

Phase-5 gate:
