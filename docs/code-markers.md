# Code markers (`HARDxx-NNN`, `BUG-NNN`)

Throughout the codebase, comments tag a fix with a marker such as `HARD19-022:` or
`BUG-066:`. Each marker is one finding from a review-and-harden cycle (`HARDxx-*`) or an
exploratory bug-hunt (`BUG-*`). The authoritative per-finding write-ups (failure scenario,
code anchors, fix sketch, regression-test strategy) live in the working docs under
`docs/plans/hardening/<cycle>/review.md` and `docs/agentic/BUGS_FOUND.md`, which are
intentionally **git-ignored** (large, process-internal, and superseded each cycle).

This file exists so a marker in the code is never a dangling reference: every marker that
appears in a committed source comment is listed below with a short account of the bug (what
went wrong) and the fix (what changed), so a developer has enough context to understand a
code comment without needing the git-ignored working docs. For deeper detail, use
`git log --grep=HARD19-022` (each fix commit carries its marker and a full summary) or, if
present locally, the working doc for that cycle.

## How this file is maintained

Regenerate the entries from the (local, git-ignored) working docs so it stays in sync with
the markers actually referenced in committed code — content sourced from each cycle's
`review.md` failure-scenario/fix-sketch write-ups, the actual fix commit messages
(`git log --grep=<marker>`), and `docs/agentic/BUGS_FOUND.md` for `BUG-*` markers. Only
markers referenced in `src/` or `android/` are included (this is a reference for code
comments, not a full backlog).

## Hardening findings

### `HARD9-001` — Wrong or changed device password strands the app in OFFLINE with no password prompt and no visible reason

**Issue:** When a user changed or enabled the network password on the C64U (or secure storage lost the stored one, or a corrupted password got saved), every discovery probe on the next app start received HTTP 403, but probes ran with `__c64uIntent: "system"` which auto-suppressed the global auth challenge and `probeOnce` did not record HTTP errors into `lastProbeError`. Discovery then expired into `OFFLINE_NO_DEMO`, and in the OFFLINE/CONSERVATIVE state machine background probes and user CTAs were both blocked before any HTTP could fire, so nothing could ever produce the 403 that would raise `notifyAuthRequired`. The app therefore showed a plain `OFFLINE` badge indefinitely with no reason and no password prompt — observed live on hardware where a c64u with password `pwd` and a saved wrong password `pwdt` showed only a generic `OFFLINE ○` badge while `X-Password: pwdt` requests failed in logcat, leaving re-entering the password by guesswork as the only recovery.

**Fix:** In commit 53eda43f, `probeOnce` now detects 401/403 via `isAuthRequiredError`, records `lastProbeError` as "Password required", and raises the app-wide `notifyAuthRequired` dialog even though system discovery probes normally suppress the request-layer auth popup. The startup discovery-window expiry was changed to preserve an auth-required probe failure instead of overwriting it with the automatic LAN-discovery fallback, so the user sees the password prompt and reason while the connection state settles to `OFFLINE_NO_DEMO`. Regression coverage was added for both the direct 403 probe path and the full startup discovery path where every probe is rejected for auth, while non-auth HTTP failures still do not raise the password prompt.

### `HARD9-004` — Saved-password editing is implicit — masked field silently mutates and persists the secret before verification

**Issue:** The "Network Password" masked input was populated with the real decrypted saved password, so the masked dots looked like a placeholder and a stray keystroke appended to the actual secret (for example "pwd" becoming "pwdt") with no reveal toggle, dirty indicator, or confirm step. On "Save & Connect", `evaluateNewDeviceReachability` classified 401/403 as `needs-password` which counted as save-allowed, so the wrong password passed the "never persist an unreachable device" gate and `setPasswordForDevice(...)` persisted the corrupted secret before `switchSavedDevice` verification even ran. This exact sequence happened live on hardware, and its only later symptom was a generic OFFLINE state (feeding directly into HARD9-001), meaning the user silently corrupted their own stored credential and was then stranded with no explanation.

**Fix:** In commit df7ddcd9, `passwordInput`/`passwordInputRef` no longer load the stored secret; a saved password renders as a disabled dots field with a "Change" button that starts an empty draft, and `handleSaveConnection` reuses the untouched stored password (fetched only at save time, never rendered) when the user hasn't clicked Change. A `needs-password` reachability result while `passwordEditing` is true and a password was supplied is now treated as a wrong-password auth failure that blocks the save entirely, so bad credentials are never written. The fix is partial in scope: because `switchSavedDevice`'s own post-persist verification reads the saved-device record from persisted storage, the persist must still precede that call, so the pre-persist reachability gate (now also catching wrong passwords) is what prevents saving bad credentials, leaving only a much narrower same-millisecond device-side password-change race as a theoretical remainder.

### `HARD9-005` — Duration override slider silently clobbers every playlist item's resolved duration

**Issue:** When a playlist had songlength-resolved durations (from HVSC or Songlengths.md5) and the user dragged the "Default duration" slider or edited the input, the 500ms-debounced `applyDurationOverrideToPlaylist` overwrote `durationMs` on every item — including items that already carried correct resolved songlengths. The clobbered values were then persisted via `durationOverrideMs`/`defaultDurationMs`, so totals, per-row durations, and auto-advance timing all went wrong. Because the enrichment effect only filled `null` durations, resolved values never came back except for SID items relaunched via `startPlaylist`, and MOD/PRG/CRT items lost their real durations permanently, which is a genuine data-loss bug on curated playlists.

**Fix:** Commit 76cb69da added an optional `durationSource?: "default"` marker to `PlaylistItem`/`PlayableEntry` and the IndexedDB `TrackRecord`, set only by `applyDurationOverrideToPlaylist`. The function now only fills items that have no duration yet or whose current duration was itself a prior default write, so a genuinely resolved duration (from songlengths, a SID header, an HVSC md5 match, or anything set outside the override path) is never touched. The marker is threaded through the existing persistence plumbing so it survives reload, and `playlistItemDuration`'s existing runtime fallback (`item.durationMs ?? durationFallbackMs`) was left unchanged since it already covered unresolved items independently of this write path.

### `HARD9-006` — Changing duration mid-track does not re-arm the auto-advance guard

**Issue:** When a track was playing with a duration of 3:00 (so `dueAtMs = start + 180000`) and the user dragged the duration to 10:00, the progress bar showed 10:00 but `autoAdvanceGuardRef.current.dueAtMs` was never updated, so `syncPlaybackTimeline` still auto-advanced at the old 3:00 mark. The reverse case was equally broken: shortening 10:00 to 1:00 pinned the progress bar at 100% / 0:00 remaining while the track actually kept playing for nine more minutes. The only writers of `dueAtMs` were `playItem`, pause/resume, and session restore, whereas the duration handlers updated only `durationMs` state, so any mid-track duration edit desynced the on-screen timeline, the auto-advance timing, and the native background watchdog (which received the same stale `autoAdvanceDueAtMs`).

**Fix:** Commit e3fdff49 added a pure `resolveAutoAdvanceDueAtMsOnDurationChange` helper in `playbackGuards.ts` plus a `PlayFilesPage` effect watching `durationMs` that recomputes `guard.dueAtMs = trackStartedAtRef.current + durationMs`. The helper returns null (a no-op) while paused, because `handlePauseResume` already recomputes `dueAtMs` from the live `durationMs`/`elapsedMs` on resume, and it is also a no-op on ordinary track launches since `playItem` already sets `trackStartedAtRef`, `guard.dueAtMs`, and `durationMs` in sync. Re-arming flows through the existing `setAutoAdvanceDueAtMs` path, so the recomputed value is also pushed to the native background watchdog.

### `HARD9-007` — Shuffle checkbox does not shuffle playback; Reshuffle irreversibly destroys curated order

**Issue:** Checking the "Shuffle" box did not randomize playback at all — `handleNext` and auto-advance still walked `activeIndex + 1`, and the checkbox only enabled the Reshuffle button. Pressing Reshuffle physically reordered the playlist array and persisted it, and unchecking Shuffle did not restore the original order, so the user's curated ordering was destroyed permanently. The persistence layer even shipped never-used `randomSeed`/`randomCursor` fields and an unused `RandomPlaySession` repository machinery that was clearly intended for a non-destructive shuffle that had never been wired up, so the feature was both non-functional and actively data-destroying.

**Fix:** Commit 9fdc7ca9 reimplemented shuffle as a non-destructive playback-order layer: `usePlaylistManager` holds a `shuffleSeed` (lazily generated on first enable, with Reshuffle drawing a new seed and no longer touching the playlist array). New pure `resolveNextPlaylistIndex`/`resolvePreviousPlaylistIndex` helpers compute a deterministic seeded shuffle order over the current item ids and walk it forward and backward, falling back to the exact prior linear-traversal contract when shuffle is off, and `handleNext`, `handlePrevious`, and `flushPendingUserSkip`'s mid-race recompute all route through them. The seed is persisted into the already-existing but previously always-null `PlaylistSessionRecord.randomSeed` field rather than adding the heavier async `RandomPlaySession` CRUD machinery, and the destructive `reshufflePlaylist`/`handleReshufflePlaylist` array-reorder helpers were removed along with the tests that encoded their destructive behavior.

### `HARD9-008` — Songlengths bundle cache cleared on every playlist change → repeated multi-MB re-parses

**Issue:** With a Songlengths.md5 loaded (up to 6MiB auto-loaded), every playlist identity change — every track transition, every duration-slider persist, every add batch — first cleared `songlengthsCacheRef` and then re-ran `applySonglengthsToItems` over the entire playlist, re-parsing the multi-megabyte database once per distinct folder. On device hardware this meant seconds of main-thread work on every auto-advance, and the same full pass inside `startPlaylist` delayed playback start on every row tap. The blanket clear was redundant because the bundle already carried a `path:mtime` signature that detected staleness, and the enrichment effect wastefully iterated all SID items even when their `durationMs` was already resolved.

**Fix:** Commit 51a5a117 removed the `[playlist, songlengthsFiles]` cache-clearing effect entirely, relying on `loadSonglengthBundleForPath`'s existing path:mtime signature check to invalidate the right cache entries only when songlengths files actually change. `applySonglengthsToItems` now skips items that already have a resolved duration, except items tagged `durationSource: "default"` (the HARD9-005 fallback marker) which stay eligible so they can still pick up a real duration later, and a successful resolution clears that marker so a later slider drag doesn't re-clobber it. The change benefits both the PlayFilesPage enrichment effect and the full-playlist re-resolution pass inside `startPlaylist`.

### `HARD9-009` — Full-range custom snapshot ($0000-$FFFF) silently saves an empty snapshot (u16 length truncation)

**Issue:** When a user picked Save RAM → Custom and entered the full range 0000–FFFF, `length = 0x10000` passed validation and `dumpRamRanges`, but `writeUint16LE(view, descriptorOffset + 2, range.length)` truncated 0x10000 to 0 because the descriptor length field is only 16 bits. The 64KiB of data was still appended, but on decode `ranges[0].length === 0`, so restore wrote nothing while still toasting "Snapshot restored". This was a completely silent data-loss bug, discovered only when the user later actually needed the snapshot, and the CPU-snapshot path already worked around this exact overflow while the custom-range path had no guard or split.

**Fix:** Commit 247dd9cf implemented two guards at different layers. `validateCustomSnapshotRanges` now splits any range over 0xFFFF bytes into safe sub-ranges — a general split rather than the CPU path's stack-page-specific three-way split, since that reasoning does not apply to arbitrary custom ranges. As a defensive backstop at the format's actual writable boundary, `encodeSnapshot` now throws on any range outside 1–65535 bytes, so this class of silent truncation cannot recur even from a future caller mistake.

### `HARD9-010` — Mount/eject success judged by HTTP status only — firmware in-body errors discarded, false "Disk mounted" toasts

**Issue:** The C64U firmware reports many failures as HTTP 200 with a non-empty `errors` array in the body, a fact the codebase already relied on elsewhere (`assertConfigWriteAccepted` and `restoreCart.ts`'s `writeWithRetry` both check `result.errors`). The `mountDrive`, `mountDriveUpload`, and `unmountDrive` methods returned `{ errors: string[] }`, but no caller inspected it, so an in-body rejected mount still produced the "Disk mounted" toast, set the optimistic `mountedByDrive[drive]` override, and cleared `driveErrors`. The user was shown success while the drive was actually unchanged, with no error surfaced anywhere.

**Fix:** Commit c734dd1d implemented the fix sketch exactly by adding `assertDriveWriteAccepted` (mirroring the existing `assertConfigWriteAccepted`) and calling it from all three drive write methods right after their response is parsed and before returning, throwing with the joined firmware errors. Because `HomeDiskManager.tsx`'s `handleMountDisk`/`handleEject` and `diskMount.ts`'s `mountDiskToDrive` already had correct catch/`reportUserError`/`driveErrors` handling in place, they simply never received a rejection to catch before this change. No UI-layer changes were therefore needed; the fix only had to make the API layer throw on in-body firmware errors.

### `HARD9-011` — CommoServe/runtime-file disks become permanently unmountable after page remount, with a misleading error

**Issue:** Disks imported from CommoServe held their bytes only as `File` objects in the in-memory `runtimeFiles` React state, while the persisted `DiskEntry` had location "local", a `sourceId` equal to the archive id, and no `localUri`. On navigating away or restarting the app, `setRuntimeFiles({})` and the module-level map were memory-only and were lost, yet the library still listed the disk. Attempting to mount then threw "Local disk access is missing. Re-add the folder or file to refresh permissions." — a message meaningless for an archive download — so the bytes were gone and the entry became a dead tombstone with a misleading error.

**Fix:** Commit 63347046 landed only a partial fix: it added `DiskEntry.sourceKind: "commoserve" | null`, set at the CommoServe import site and checked in `resolveLocalDiskBlob`'s final fallback so it throws the accurate "This disk's data is no longer available. Re-import it from CommoServe to mount it again." message instead of the generic local-folder one. The underlying data loss (archive bytes are still never persisted) was intentionally left open here because a real fix requires either persisting archive-imported bytes or re-downloading on demand, both materially larger changes needing product sign-off on the UX. The full fix landed in round 10 as HARD10-002, using re-download on demand via a persisted `archiveRef` mirroring the playlist pattern.

### `HARD9-012` — Disks page hardcodes every mount as read-only, breaking in-game disk saves

**Issue:** `handleMountDisk` always passed `{ mode: "readonly" }` (introduced incidentally in a test-hardening PR), including for the user's own D64s on the C64U, so games saving high scores or states failed with DOS error 26 "WRITE PROTECT ON". The same disk launched via Play mounted `"readwrite"`, since `mountDiskToDrive` defaults to read-write, making the two paths silently inconsistent. No UI ever exposed the mount mode, so the DOS 26 failure was inexplicable to the user, who had no way to know their library mounts were forced read-only.

**Fix:** Commit d8ed34c4 took the first fix-sketch option and removed the `{ mode: "readonly" }` override entirely, so library mounts fall back to `mountDiskToDrive`'s existing `readwrite` default and now match Play exactly. No read-only toggle was added, because nothing in the codebase or its history indicated that read-only-by-default was an intentional product decision — it traced to a single line added incidentally in a large unrelated hardening commit. This restores in-game saves for library-mounted disks with no new UI surface.

### `HARD9-013` — HVSC ingest promise never settles after explicit cancel (reject skipped in cancelled coroutine)

**Issue:** When the user tapped Cancel during an HVSC install, `cancelIngestion` both set the cancel token and called `activeJob?.cancel(...)`, so the extractor threw `CancellationException` and the catch block ran, but its `withContext(Dispatchers.Main) { call.reject(...) }` executed inside a now-cancelled coroutine. Because `withContext` checks the caller's Job on entry and throws without running its block, the reject was never delivered, so the awaited `ingestHvsc` promise hung forever and left the pipeline stuck in EXTRACTING until app restart. The same skip applied to the generic error path when cancelled and to `handleOnDestroy`, and the memory-pressure cancellation path masked the bug in testing because it only sets an AtomicBoolean and does not cancel the job.

**Fix:** Commit 74eab854 implemented the fix sketch's primary option, wrapping `call.reject(...)` in both the `CancellationException` catch and the generic `Exception` catch in `withContext(NonCancellable + Dispatchers.Main)` so the reject is delivered even from a cancelled job. `handleOnDestroy` needed no separate change because it only cancels the job and relies on the running coroutine's own (now-fixed) catch block to reject. The fix was validated via the full `:app:testDebugUnitTest` suite plus two isolated coroutine-semantics tests proving that a cancelled Job skips a bare `withContext(Dispatchers.Main)` block but still runs one wrapped in `NonCancellable`.

### `HARD9-014` — Stale "update applied" records survive HVSC library reset and permanently block updates

**Issue:** When the user tapped "Reset HVSC library data" and then reinstalled, the baseline ingested but every incremental update version was skipped with "Update N already applied", because the per-version success records from the previous install were never cleared. Since skips do not bump `installedVersion`, it stayed at baseline and `checkForHvscUpdates` reported the same updates as required forever, so the library permanently missed update content while the UI perpetually showed updates available. The root cause was that `resetHvscLibraryData` patched state without touching the `updates` record map, and `updateHvscState` preserves `current.updates` whenever the patch omits it.

**Fix:** Commit e4ad9eb2 implemented the fix sketch exactly, adding `updates: {}` to `resetHvscLibraryData`'s patch and also to `applyIngestionSuccess`'s patch when `plan.type === "baseline"`, so a direct baseline reinstall is covered too, not just the explicit-reset path. The work also surfaced that `resetHvscLibraryData` had zero prior test coverage and that the test file's `hvscFilesystem`/`hvscStatusStore` mocks were missing `resetHvscCache`/`getDefaultHvscStatusSummary` entirely, which would have thrown the moment any test called them. Those mocks were completed to make the new regression tests possible.

### `HARD9-015` — Poisoned empty browse snapshot makes recursive HVSC "add folder" silently return zero songs

**Issue:** On native SQLite-backed installs the integrity stat-probe fails because SIDs are not individual files, so `ensureHvscIndexReady` clears the browse snapshot, and `queryFolderPage` then rebuilds `this.browseSnapshot` from an empty `entriesSnapshot` and caches the empty-but-truthy snapshot. Browsing still worked via the runtime fallback, but "add all from HVSC root" went through `getHvscSongsRecursive` into the poisoned snapshot, so `querySongsRecursive("/")` returned `[]`. The adapter's `if (bulkSongs)` truthiness check treated the empty array as success and never fell back to paged BFS, so the operation silently added zero items with no error shown to the user.

**Fix:** Commit 7008d654 changed `listSongsRecursiveFromBrowseIndex` and `streamSongsRecursiveFromBrowseIndex` (the latter also used by the "add all recursively" playlist flow) to return `null` when the snapshot has zero songs, closing the loophole where the root path always exists even in a wholly-empty snapshot. `HvscMediaIndexAdapter.queryFolderPage` no longer caches its fallback rebuild into `this.browseSnapshot` when the rebuild saw zero entries, so an unloaded snapshot is retried on the next call instead of being poisoned permanently. The adapter's `if (bulkSongs)` check needed no change since it already falls back to BFS on `null` and now only sees `[]` for a genuinely-empty query within a populated library, and new tests confirmed the failure against pre-fix code.

### `HARD9-016` — Latest-intent write lane silently drops committed writes to a different config item

**Issue:** A single `useInteractiveConfigWrite` lane was shared by all eight SID sliders (4 SIDs times volume/pan) and both lighting sliders, and the lane held a single `latestJob` that `schedule()` replaced regardless of which config item the payload targeted. If the user committed SID1 volume then SID1 pan within the 400ms quiet window (or dragged both), the queued volume update was replaced and never sent to the device. Worse, `resolveUpTo(settledVersion)` resolved the dropped job's promise as success, so `useDeviceBoundSlider` latched a pendingIntent the device never echoed, and after the 2–4.5s watchdog the slider silently snapped back with no error or toast — a genuine data-loss bug on config writes.

**Fix:** Commit ed0d8d3f added an optional `merge` function to `createLatestIntentWriteLane` (defaulting to replace, preserving the lane's plain string/number/`PlaybackMixerWrite` use elsewhere) and wired `useInteractiveConfigWrite` to merge per-item updates via object spread. Merging only at `schedule()` was insufficient because `process()` pulls the first job out synchronously before its first await, so the same merge was also applied at the two later points where a newer job supersedes one already waiting in `beforeRun()` or one whose `run()` just failed. The pre-existing volume/pan preview-mode asymmetry (volume `commitOnly`, pan defaulting `throttled`) was documented as intentional at the `AudioMixer.tsx` call site since both now safely share the merging lane, and new tests were proven failing against pre-fix code.

### `HARD9-017` — Profile load / revert / load-from-flash never invalidate `c64-config-items` — Home shows pre-load values

**Issue:** When the user tapped "Load From App", "Revert", or "Load From Flash", a toast said the config had loaded, but every quick-config control on Home (Turbo, Video Mode, RAM Expansion, SID cards, lighting) kept showing the old values, because those controls read through `useC64ConfigItems` on the `["c64-config-items", ...]` key with a 30-second staleTime and nothing invalidated that key family. There was a reverse gap too: Config-page writes invalidated only `c64-category`/`c64-all-config`, so Home stayed stale for up to 30 seconds after any write. The result was that Home routinely displayed config values that no longer matched the device or the just-loaded profile.

**Fix:** Commit d6fe352b added `c64-config-items`/`c64-config-item` invalidation to all three gaps: a blanket invalidation in `applyConfigData` (since a profile load or revert spans every category in the snapshot) and a category-scoped invalidation in `useC64SetConfig`/`useC64UpdateConfigBatch`'s `onSuccess` matching their existing `c64-category` invalidation. Prefix invalidation already worked because the routing epoch is appended to the key. The `useC64UpdateConfigBatch` `skipInvalidation` branch (the interactive slider write lane) was deliberately left untouched, since it already has its own debounced reconciliation invalidation and adding this there would defeat its purpose of avoiding a refetch storm during drags.

### `HARD9-018` — Save-to-App, initial revert baseline, and revert "verification" read a persistent client cache instead of the device

**Issue:** The idle initial snapshot (5s after connect) captured categories via `getCachedCategory`, which served `selected` values from a localStorage-persisted, no-TTL enrichment cache — effectively last session's values for categories not yet read this session — so if settings had changed via the C64U's own menu, the baseline was wrong and "Revert Changes" would actively write stale last-session values back to the device. "Save to App" snapshotted the same stale cache. Additionally, `revertToInitial`'s "verification" refetched through the same cache that `updateConfigBatch` had just updated per-item via `setCachedConfigValue`, so verification compared the writes against themselves and always passed even if the firmware had clamped the values.

**Fix:** Commit f579fc56 established that all four `fetchAllConfig` call sites (idle initial snapshot, manual initial snapshot, Save-to-App, revert verification) need a ground-truth read and none want cache-serving, so rather than thread a `fresh` flag it removed the `getCachedCategory` shortcut from `readCategorySnapshot` entirely and deleted the now-unused `getCachedCategory` method from `C64API`. The distinct, still-used `getCachedConfigItem` cache for static option metadata was left untouched, matching "keep the cache for background enrichment only." Tests that exercised the removed method or encoded the old stale-cache reuse were updated or removed, and a new test proves every category is read fresh regardless of what is cached. This closed the gap on every one of the four call sites in a single pass rather than fixing them piecemeal.

### `HARD9-019` — Full trace ZIP exported on every recorded error

**Issue:** When the device was unreachable — common given c64u flakiness — every failed poll produced a fresh `Error`, so `errorOnce` (a WeakSet keyed by Error identity) never deduped, and each failure ran `exportTraceZip()`, doing a `JSON.stringify` over up to 25,000 events / 50MB plus a synchronous `zipSync`, all on the main thread every few seconds for the entire outage. This caused UI jank, battery drain, and GC churn precisely when the app was already degraded. The eager export ran on every non-OK HTTP response, every non-expected network failure, every window error or unhandled rejection, and every failed traced action.

**Fix:** Commit 4c292809 removed the background export entirely rather than debouncing it, because `getLastTraceExport()` and the `"c64u-trace-exported"` event it dispatched had no callers or listeners anywhere in the app, and the real Share/Download Diagnostics flow always calls `exportTraceZip()` itself, fresh and on demand. The eager per-error export was therefore pure wasted work with no functional purpose to preserve. The error event itself is still recorded via `appendEvent`, and the duplicate-Error-instance dedup guard was left unchanged.

### `HARD9-020` — Unguarded localStorage log writes; O(n) parse/stringify per log line

**Issue:** `writeLogs` had no try/catch, so on a localStorage quota-exceeded error `addLog` would throw, and because the console bridge routes every `console.warn`/`console.error` through `logger.warn`/`logger.error` into `addLog`, any app code calling `console.warn` could then throw unexpectedly. Separately, `addErrorLog("Unhandled promise rejection", { reason: event.reason })` with a circular `reason` would throw inside the rejection handler itself. On top of these robustness holes, every single log line did a JSON.parse of the entire 500-entry blob plus a full re-stringify plus a synchronous `setItem`, so each log line was O(store) main-thread work, amplified by the console bridge.

**Fix:** Commit 24cd3ce4 implemented all three parts of the fix sketch: `persistLogsNow` wraps the write in try/catch, halving and retrying once on failure and warning (never throwing) on a second failure. `addLog` sanitizes `details` through a circular-safe JSON round-trip at capture time, protecting every later serialization point rather than just the immediate write. Reads and writes now go through an in-memory cache with 500ms debounced persistence instead of a full parse/stringify per call, and the fix switched from `window.setTimeout` to the global `setTimeout`/`clearTimeout` after discovering some test environments provide a minimal `window` without timer methods.

### `HARD9-021` — Closed diagnostics overlay copies full trace/log stores on every event

**Issue:** `GlobalDiagnosticsOverlay` is always mounted, and every trace event append dispatched `c64u-traces-updated`, which called `setTraceEvents(getTraceEvents())` and copied the entire up-to-25,000-element array and re-rendered the overlay even while it was closed, so a single REST poll emitting ~3 events meant 3 full copies and renders. Every `addLog` similarly dispatched an update that called both `setLogs(getLogs())` and `setErrorLogs(getErrorLogs())`, each re-parsing the full 500-entry localStorage blob (and `getErrorLogs` calls `getLogs` again), so one log line cost three full JSON parses. Idle polling alone therefore drove continuous O(25k) main-thread work over long sessions, entirely for an overlay the user was not even looking at.

**Fix:** Commit 7a72f6e7 changed both listener-registration effects (logs/error-logs and trace events) to subscribe only while `overlayOpen` is true and to unsubscribe on close, seeding a fresh snapshot on open, so the expensive copies happen only when the overlay is actually visible. The separate `getErrorLogs` re-parsing concern was already resolved by HARD9-020's in-memory log cache that landed earlier in the same session, so no additional change was needed for that half of the evidence. The net effect is that a closed diagnostics overlay does no per-event work at all.

### `HARD9-025` — Password-field write race between connection state and secure-storage load

**Issue:** Two independent effects owned `passwordInput`: one mirrored `password` from `useC64Connection` on deps `[password]`, and the other async-loaded the selected device's stored password. While the user was typing a new password, any runtime connection-config change could silently replace the in-progress input, so Save would persist the old or mixed value read from `passwordInputRef`. Switching devices mid-flight could also land a stale password into the field for the new device, because both sync effects wrote `setPasswordInput`/`passwordInputRef.current` with no dirty guard.

**Fix:** Commit df7ddcd9 (the same commit as HARD9-004) removed both racing effects entirely rather than dirty-guarding them, because the HARD9-004 fix means the field never displays the real secret and so the `[password]`-mirroring effect and the async `getPasswordForDevice` preload effect were no longer needed. The only remaining effect resets the field on `[selectedSavedDevice?.id, selectedSavedDevice?.hasPassword]`, i.e. an actual device switch, which is the one case where resetting the draft is correct. There is therefore no longer any effect that can overwrite an in-progress keystroke from live connection or password state.

### `HARD9-026` — 600ms drag-settle timer collapses a held swipe

**Issue:** When the user dragged horizontally to peek at the next tab and then held still for more than 600ms, no pointermove fired, so `dragOffsetPx` stopped changing and the `phase === "dragging"` effect's timer expired and called `setRunway(buildIdleState(...))`, snapping the runway back while the finger was still down. The next movement then re-entered onProgress with the full dx, causing a visual jump, and releasing during the reset silently lost a swipe that had already crossed the commit threshold. The root cause was that the effect re-armed on `[centerIndex, dragOffsetPx, phase]`, so it stayed alive only while dx changed, making a stationary held pointer indistinguishable from the missed-pointerup case the timer was originally added to handle.

### `HARD9-034` — Concurrent playlist repository commits can persist a stale snapshot

**Issue:** When two playlist changes happened in quick succession (for example a `playItem` duration write followed by a user removal), the persist effect launched commit A (v1) then commit B (v2), and because `inflightCommits` only deduped identical snapshot keys, both ran concurrently. ### `HARD9-035` — CPU snapshot "saved" toast can fire while the C64 is left frozen — resume failure swallowed despite the comment saying it must be surfaced

**Issue:** After a successful register capture, the program is frozen in the handler spin loop, and if `resumeAfterCapture` failed (for example a transient c64u drop mid-writemem), the `finally` block did `.catch((error) => { addErrorLog(...) })`, swallowing the failure. `captureCpuSnapshotData` then returned normally, so the user saw "CPU + RAM snapshot saved" while their C64 actually sat frozen with the IRQ vector still pointed at the capture handler. The bug was especially egregious because the comment directly above the swallow explicitly stated the resume failure "must be surfaced (not swallowed) so the UI can offer a manual recovery path (Restore / power-cycle)."

**Fix:** Commit 60775949 took the "return it" half of the fix sketch rather than "rethrow", because throwing would discard the successfully-captured and separately valid snapshot data via the caller's try/catch when capture and resume are genuinely independent outcomes. It added a `resumeError: Error | null` field threaded through `captureCpuSnapshotData` into `createCpuSnapshot` and then `handleSaveCpuSnapshot`. That handler now shows a destructive toast naming the underlying error and telling the user the snapshot was saved but the program could not be resumed, instead of the bare success toast.

### `HARD9-036` — CPU restore failures strand the machine in the restore cart and never fall back to RAM-only restore

**Issue:** `CpuRestoreUnsupportedError` was documented as a case where the caller should offer RAM-only restore, but `handleRestoreSnapshot` caught nothing, so the user just got a raw "stack pointer $f is below the safe minimum…" toast with no fallback. Worse, if the handshake failed after `runCartridgeUpload` (a READY timeout, or a write failing all four retries mid-DMA), the C64 had already been reset into the uploaded spin-loop cart and was frozen, yet the error path performed no recovery at all — no reset, no release-flag retry, and no guidance. The machine could therefore be left wedged in the restore cartridge with the user given no explanation and no path forward.

**Fix:** Commit f856dfee implemented case (a) exactly: `handleRestoreSnapshot` now catches `CpuRestoreUnsupportedError` and falls back to `loadMemoryRanges`, toasting why exact CPU state could not be restored. For case (b) it chose `machineReset` over writing the RESTORE_FLAG_GO release flag, because releasing the cart when the RAM/RTI-frame writes never completed would finalize into an incomplete RTI frame and likely crash into garbage, whereas a full reset recovers correctly regardless of which post-upload step failed. `restoreCpuSnapshot` now wraps everything after the cartridge upload in a try/catch that attempts the reset best-effort and rethrows an error whose message states the outcome (either the machine was automatically reset, or the user must power-cycle it manually).

### `HARD9-037` — Mount sheet permits concurrent mounts to the same drive

### `HARD9-038` — Local-disk group rotation and eject-before-delete break as soon as the first drives poll lands

### `HARD9-039` — Capture-timeout rollback can corrupt the safe region while the CPU is executing it (late-interrupt race)

**Issue:** The capture poll loop checks a `captured` flag and, on deadline, pauses and rolls back by restoring the vector, restoring the safe region, and trying the next candidate. If the program's IRQ fired between the final `captured` read and `machinePause()` (a window of up to 50ms plus latency), the CPU would be frozen inside the handler's spin loop at $033C, and the rollback would then overwrite that spin-loop code under the CPU's feet. On resume the CPU would execute arbitrary restored bytes at its current PC, crashing the very program the user asked to snapshot, because the timeout path never re-checked `layout.captured` after pausing.

**Fix:** Commit 42421f0b implemented the fix sketch exactly by splitting the single `if (!captured)` block into two: the first calls `machinePause()` and re-reads the flag, and the second — the actual rollback plus `continue` — only runs if the flag is still false after that re-check. A flipped flag now falls through unchanged into the existing "captured" handling (stable-read verification and so on), with no duplicated logic. This closes the late-interrupt race so a capture that succeeded at the last instant is treated as a success rather than being destructively rolled back.

### `HARD9-040` — Failed HVSC baseline promotion can delete the only copy of the user's library

**Issue:** During a baseline ingest (`resetLibrary=true`), the metadata DB was wiped, repopulated, and committed before the directory swap. If `libraryRoot.renameTo(oldRoot)` succeeded but `stagingRoot.renameTo(libraryRoot)` failed, the recovery `oldRoot.renameTo(libraryRoot)` could also fail, and the exception reaching the catch block would then unconditionally `oldRoot.deleteRecursively()`, permanently deleting the sole remaining copy of the user's library. Even when recovery succeeded, the DB had already been committed against files that were deleted with staging, so every HVSC entry would 404 until a full re-ingest — a serious data-loss and consistency hazard.

**Fix:** Commit d8408eda extracted the promotion sequence into `promoteBaselineLibrary`, running the directory swap first and only committing the DB rewrite once the swapped files are in place; if the DB write then fails, the swap is rolled back best-effort so the never-committed DB keeps matching what is actually on disk. New reflection-based tests exercise the happy path, first-rename failure, second-rename failure with successful recovery, and the guard's core invariant, all proven failing against pre-fix code.

### `HARD9-041` — Late setDueAtMs resurrects a phantom foreground service + wake lock after Stop

**Issue:** The generation gate only protected the `dueAtMs == null` clear and stale starts, so if a non-null due update landed after `stop()` — which happens because the Play page pushes dueAt through an async latest-intent lane where a queued auto-advance write can flush after Stop — `updateDueAt` fell through to `startForegroundService(ACTION_UPDATE_DUE_AT)` carrying the current post-stop `commandGeneration`, which was accepted. **Fix:** Commit 97a79ce5 applied the fix sketch's parenthetical, unifying the non-null branch onto the same `startPendingGeneration` check the null (clear) branch already used, instead of an unguarded second intent-send path keyed on the current post-stop `commandGeneration`. ### `HARD9-042` — Stale-generation FGS start intent stops itself without ever calling startForeground (crash risk)

**Issue:** When JS `start()` queued a `startForegroundService` at generation N and `stop()` bumped to N+1 before `onStartCommand` ran (a fast play-then-stop under auto-advance churn), the stale branch called `stopSelf(startId)` without ever calling `startForeground()`. This risks the `RemoteServiceException: Context.startForegroundService() did not then call Service.startForeground()` crash, because the stop-before-timeout waiver in ActiveServices is racy and widely reported on Android 8–13. The same hazard applied to the `intent == null` sticky-restart path, since `startForeground` was only reached in the `!isRunning` start branch.

**Fix:** Commit 065137fc applied the fix sketch via a shared `satisfyForegroundContractAndStop` helper called from both the null-intent and stale-generation branches, which calls `startForeground` then immediately `stopForeground` plus `stopSelf`. It added one safety refinement beyond the literal sketch: the throwaway dance only runs when `!isRunning`, because if the service is already legitimately in the foreground under a newer generation, unconditionally calling `stopForeground` would tear down that still-active notification, so a stale intent for an already-running service stays a pure `stopSelf(startId)` no-op. `stopForegroundCompat` branches on `Build.VERSION.SDK_INT` since minSdk 22 predates the `STOP_FOREGROUND_REMOVE` int-flag overload, and regression tests prove startForeground fires before stopping in both branches while an already-running service's notification is left untouched.

### `HARD9-044` — FTP/SAF readFile buffers entire file + Base64 in heap — OOM on large files

**Issue:** `FtpClientPlugin.readFile` accumulated the whole transfer in a `ByteArrayOutputStream`, then copied it with `toByteArray()`, then Base64-encoded it (~1.33x), for a peak of roughly 3.3x the file size plus the bridge JSON. `FolderPickerPlugin.readFile`/`readFileFromTree` did the same, and `pickFile` accepted `*/*`, so tapping a large file such as a .dnp disk pack or a firmware image drove the app into OOM — a hard crash, possibly mid-playback. There was no size cap anywhere in either plugin, so nothing bounded the memory a single read could consume.

**Fix:** Commit b0b9941c applied the first fix-sketch option, a max-size guard rather than a streaming-to-disk rewrite. `FtpClientPlugin.readFile` now rejects immediately when a caller-declared `totalBytes` exceeds 32MB and also checks the running byte count inside the existing chunked read loop, so an absent or wrong hint cannot bypass the guard, while `FolderPickerPlugin` replaced both unbounded `readBytes()` calls with a shared bounded-read helper checking the same 32MB cap during the read. Both caps are `internal var` for test overridability, and regression tests cover the upfront and incremental FTP guards and the SAF `readFile` guard (a symmetrical `readFileFromTree` test was dropped because SAF tree-URI resolution does not work against a plain `file://` URI in the test harness, and the guard logic is shared via the same already-covered helper).

### `HARD9-045` — normalizeSourcePath collapses internal whitespace, corrupting legitimate paths on every source

### `HARD9-046` — Ingestion finalize overwrites the songlengths projection with duration-less, metadata-stripped song records

### `HARD9-047` — Web/desktop local sources list files after reload that can no longer be opened

### `HARD9-048` — Disk library save effect writes stale/empty state before the load effect settles

**Issue:** On mount and on every `uniqueId` change, the persist effect (`saveDiskLibrary(uniqueId, { disks })`) ran in the same commit as the load effect, before the loaded disks re-rendered, so it wrote `[]` on first mount or the previous device's list under the new device's key during an A-to-B switch. ### `HARD9-050` — Throttled-preview sliders leave the device at an intermediate value when the user returns to the start position

### `HARD9-051` — Home quick-config writes never set hasChanges, so "Revert Changes" stays disabled

### `HARD9-058` — Fetch trace duplicates full request/response payloads in the hot path

**Issue:** Every `/v1/` request was traced by fully copying request Blobs/ArrayBuffers (`await body.arrayBuffer()` into a new `Uint8Array`) just to build a preview, so uploading a multi-MB disk image doubled it in memory; every response was `clone()`d and fully read, and for JSON the entire parsed body was stored in the trace event, keeping whole config payloads resident in the 25k-event store. ### `HARD9-067` — Snapshot restore halts the CIA TOD clocks and can flip interrupt-mask bits (ICR write semantics)

**Fix:** The fix sketch was implemented exactly: `isCiaTimerRegister` (the single shared predicate both restore paths already call) was extended to also match $xx08-$xx0B and $xx0D. ### `HARD9-072` — TelnetSocket state read/written across threads without synchronization

### `HARD9-073` — cancelRead after completion leaves permanent entries in cancelledReads

### `HARD9-079` — Native CommoServe transport ignores AbortSignal — Cancel doesn't cancel

### `HARD9-083` — Pre-aborted or early-aborted FTP read still performs the full transfer

**Issue:** When a caller aborted a queued FTP read before the native call started (a user cancelling a bulk import), `executeFtpRead` saw `signal.aborted`, fired `cancelRead` (a no-op because the read was not registered yet), and then proceeded to `readFile(...)` anyway, downloading the whole file. ### `HARD9-084` — HVSC ingestion cancellation unchecked during the deletion pass and index finalize

### `HARD9-086` — Optimistic rollback can resurrect a stale pin when two rapid writes to the same item race

**Issue:** Picking value A (pin A, write A) then quickly picking B (previousEntry_B = {A}, pin B, write B queued) set up a race: if write A failed, `restoreEntry(key, undefined)` deleted the pin (flipping the UI to the stale device value while B was in flight), and if B also failed, `restoreEntry(key, {A})` re-pinned A — a value the device never accepted — latched until an accidental echo or remount. **Fix:** The fix sketch was implemented exactly: `restoreEntry` now takes a required third argument (the value this write pinned) and only applies the delete/restore if the store's current entry for that key still equals it; otherwise a newer write has taken ownership and its own success/failure handling is authoritative. New tests cover both the two-writes-racing case and the ordinary single-write rollback, proven failing against the pre-fix 2-arg signature.

### `HARD9-087` — useInteractiveConfigWrite pending/burst flags are wrong under concurrent writes

**Issue:** With two overlapping writes on one lane, the first to settle ran `finally: writeBurstActiveRef.current = false; setIsPending(false)` while the second was still in flight, so `isPending` read false during an active write. ### `HARD10-009` — IndexedDB `track:*` records are never garbage-collected

### `HARD12-008` — Kernal-mode disk autostart writes 16 bytes into the C64's 10-byte keyboard buffer, scribbling OS variables ($0281-$0286)

### `HARD12-009` — DMA disk autostart misclassifies large/nonstandard tokenised BASIC as machine code and SYSes into the program header

A false "BASIC" on an ML program with a standard BASIC stub still works, keeping the classification change low-risk.

If that batch failed with no newer job queued, the lane's failure path ran `resolveUpTo(job.version - 1)` before `rejectUpTo(job.version, err)`, so waiter v1 (slider A) resolved as success even though its value was part of the failed device write — only slider B's caller got the error toast/retry. Slider A's UI then silently snapped back to the device value on the reconciliation refetch with no error and no retry affordance, the exact silent-drop the merge was added to prevent, now on the failure side.

**Fix:** The lane now tracks which versions were merged into the running job and, on failure, rejects all of them (and on success resolves all of them), while keeping the replace-semantics behaviour for non-merge lanes where superseded values were never sent. A code comment at `latestIntentWriteLane.ts:101` records that in merge-semantics lanes every waiter up to `job.version` was part of the failed batch and must be rejected. Only the failure-path semantics change, and only for merge lanes.

### `HARD12-011` — Late /v1/info success from the previous device can stamp its identity onto the newly selected saved device

**Issue:** `setSnapshot({deviceInfo})` unconditionally attributed arriving `deviceInfo` to the currently selected saved device via `rememberSelectedSavedDeviceIdentity`, and its host guard derived from the runtime API config. During a saved-device switch the selection flips before `applyC64APIRuntimeConfig` updates the runtime host, so an in-flight background `/v1/info` poll of the old device completing in that window passed the host guard and persisted the old device's product/hostname/unique_id/firmware onto the new saved device, marking it verified. The wrong `lastKnownProduct` then fed product-family-gated config-write validation and Settings displayed the wrong identity until the next successful verification.

**Fix:** A saved-device switch probe window is now opened around `selectSavedDevice` → `applyC64APIRuntimeConfig` so identity attribution is suppressed for a late response attributed to the wrong selection, and the switch resolves the password first (HARD12-003) to shrink the window. Code comments at `connectionManager.ts:468/510` and `useSavedDeviceSwitching.ts:109` record the probe window that guards the selection against a late `/v1/info` from the old device. This same window is later reused by the HARD16-001 startup-sweep fix.

### `HARD12-012` — Legacy default password bleeds to every saved device without its own entry

**Issue:** `resolvePasswordForDevice` fell back to `legacyDefaultPassword` for any device id lacking its own entry, and the multi-device migration never cleared that legacy field, so `getPasswordForDevice` copied it into whatever device was requested while leaving `legacyDefaultPassword` set. For a user upgraded from the single-device era with password P, adding and selecting a new device caused every `getPassword` consumer — FTP transfers, Telnet login, config-file application, and the REST `X-Password` header — to transmit P to the new device's host, and the first `getPasswordForDevice` call permanently copied P into the new device's slot and flipped its `hasPassword` flag. This was both a credential-disclosure issue (device A's secret sent to host B on the LAN) and a correctness issue (auth-failure loops with no visible cause).

**Fix:** The legacy default password is now migrated exactly once — to the device selected at first multi-device load — and then cleared, and `resolvePasswordForDevice` returns `null` for devices without their own entry instead of leaking the legacy value. `getPasswordForDevice` no longer silently migrates the legacy password into an arbitrary requested device's entry. Code comments at `secureStorage.ts:73/143/241/248` document the once-only migration and the removal of the silent per-device copy, with care taken not to lose the original device's password during migration ordering.

### `HARD12-013` — Every local disk mount hashes the full disk image (up to 64 MB) for a debug log line even with debug logging disabled

**Issue:** `resolveLocalDiskBlob` computed `buildBinaryFingerprint(bytes)` — a byte-by-byte FNV-1a loop over the whole image (bounded at 64 MB) — on every resolution, purely as an argument to `addLog("debug", ...)`, which discards debug entries unless debug logging is enabled. The runtimeFile branch was worse, performing a full extra `runtimeFile.arrayBuffer()` read of the image solely to feed the fingerprint before returning the original File untouched. On mid-range Android this added a full-image copy plus a JS-loop hash to every interactive mount and every local disk-playback start — user-perceptible latency on a hot path, for a log line usually thrown away.

**Fix:** The expensive fingerprint (and the runtimeFile branch's extra read) are now gated behind `loadDebugLoggingEnabled()` so the hash runs only when the diagnostic log will actually be emitted. Code comments at `diskMount.ts:566/684` record skipping the full-image FNV-1a hash when debug logging is off, since the diagnostic log is the only consumer of the bytes. This is a logging-only change with negligible risk.

### `HARD12-014` — Non-recursive folder add still walks the whole folder tree recursively (FTP on c64u) for songlengths discovery

**Issue:** With "recurse folders" off, selecting a directory added only its direct children, but the songlengths-discovery pass still called `source.listFilesRecursive(selection.path)` for every selected directory. For an `ultimate` source this was a full recursive FTP walk of the subtree on the c64u's single-threaded, load-fragile FTP service — precisely the load the user avoided by disabling recursion, and a documented trigger class for the firmware FTP wedge — to discover songlengths files for files that were never added. Selecting a large folder (e.g. an SD-card HVSC mirror root) non-recursively could trigger minutes of FTP listing traffic.

**Fix:** The non-recursive branch now searches only the already-listed direct children (the listing cache for the selected path) for songlengths files, matching the scope of what was actually added, rather than triggering a recursive listing. A code comment at `addFileSelections.ts:908` records that when the user opts out of recursion the add limits songlengths discovery accordingly. Songlengths in deeper subfolders stop being auto-discovered for non-recursive adds, which matches the add's own scope.

### `HARD12-015` — Per-file O(n²) rebuild of the prefetched config-candidate map during playlist adds

**Issue:** `appendPlayableFile` ran for every added local/ultimate file and called `discoverConfigCandidates` with `getPrefetchedConfigEntriesByPath()`, a getter that iterated all parents seen so far and rebuilt each parent's merged entry map into a fresh `Map` on every call. Adding n files from one folder therefore cost O(n²) map insertions plus n array allocations of size n — for a 2,000-file folder roughly 4M map operations on the UI thread interleaved with the add loop, adding seconds of stall on Pixel-4-class hardware on top of the per-file config-candidate work. Both defects compounded on large libraries, since the app's own bookkeeping — not the device or network — became the bottleneck for a bulk add.

**Fix:** The per-parent prefetched entry map is now built once per parent (dirty-tracked / built once per batch) rather than rebuilt for every parent on every file, making the merge work O(n) for an n-file single-folder add. A code comment at `addFileSelections.ts:574` records building the per-parent prefetched entry map once per parent. This is a pure caching change with low risk. This turns the add-loop's cost back to linear in the number of files instead of quadratic.

### `HARD12-017` — Feature: remote keyboard/joystick input sheet using `machine:input` with a kernal keyboard-buffer fallback

**Issue:** The app had no way to relay keyboard or joystick input to the C64, and the only available injection transport (the kernal keyboard buffer) was fragile. The review proposed a Home "Remote Input" sheet that uses the U64-family `POST /v1/machine:input` batch endpoint when the firmware offers it (absent in the documented c64u 1.1.0 API, expected in an upcoming firmware) and falls back to the kernal keyboard buffer otherwise, driven by a runtime capability probe (200 = full keyboard+joystick, 501 = hardware unsupported, 404/405 = older firmware) and device-safety rate-limiting so rapid input could not flood the single-threaded firmware HTTP task. Kernal-buffer injection alone was slow, one-key-at-a-time, and offered no joystick relay at all, making direct-to-machine input a weak spot in an otherwise capable remote-control app.

**Fix:** The Remote Input feature was implemented as a bottom sheet (`RemoteInputSheet`) driven by `useRemoteInputSession`, with capability tiers derived from a `machine:input` probe (`capabilityTier.ts`), a coalescing `machineInputThrottle` for the full-tier POST transport, and a chunked kernal keyboard-buffer fallback (`kernalFallbackEncoding`/`kernalFallbackInjector`, reusing the HARD12-008 ≤10-byte chunking) with joystick disabled on that tier. The E2E `MockC64UServer` implements the `machine:input` endpoint per the u64e v3.15-alpha schema so the capable path can be tested. Code comments across `c64api.ts`, `RemoteInputSheet.tsx`, `useRemoteInputSession.ts`, `capabilityTier.ts`, and `MockC64UServer.kt` cite HARD12-017; the feature was later hardened extensively (the HARD13/HARD15/HARD16 remote-input findings).

### `HARD12-018` — Foreground service + partial wake lock retained indefinitely after a song playlist auto-ends

**Issue:** When the last playlist item is a song category (`sid`/`mod`), `finishPlaylistPlayback` deliberately keeps `isPlaying=true` so Stop stays reachable (HARD11-003), but `shouldStopBackgroundExecution` required `!isPlaying || isPaused`, so the foreground service and `PARTIAL_WAKE_LOCK` stayed held with nothing left to schedule — the auto-advance guard was null, the watchdog cleared, and no JS work remained. If the playlist ended while the phone was locked/backgrounded (the normal background-playback case), the app held the wake lock indefinitely, causing overnight battery drain for a C64 that plays its looping SID entirely on its own. Because nothing else remained to run, the service and wake lock served no purpose beyond keeping the device awake. On a phone left overnight with a looping SID playlist, this alone visibly drained the battery.

**Fix:** A `playlistEnded` input was added so background execution treats "playlist ended, guard null" as a stop condition — stopping the foreground service and wake lock — while leaving `isPlaying` (and the Stop affordance) untouched. Code comments at `backgroundExecutionPolicy.ts:9` and `usePlaybackController.ts:284/686` record tracking that the last song auto-ended and invalidating that flag when a new track starts. The change was made carefully not to regress the BUG-025/BUG-040 refcounting.

### `HARD12-019` — Discovery save can silently retarget an existing saved device when two devices share a factory hostname

**Issue:** `persistDiscoveredDevice` resolved "is this candidate an already-saved device?" by cascading matchers (uniqueId → hostname → address), and the hostname matcher did not require the uniqueId to agree. With two same-model Ultimates on the LAN sharing a factory hostname, saving discovered unit B matched saved unit A by hostname and ran `updateSavedDevice(A, {host: B-address, lastKnownUniqueId: B, ...})`, silently rewriting unit A's saved entry (host, identity, and verified summary) to point at unit B. Unit A's entry — its name, password flag, and per-device associations — then controlled the wrong physical machine, and no second device was created.

**Fix:** The hostname/address matchers now reject a match when both sides have unique ids that differ, preferring to create a new saved device rather than retarget the existing one. A code comment at `discoveryManager.ts:349` records that two devices can leave the factory sharing the same default hostname, so a differing unique id must not be deduped together. This narrows an over-eager dedupe with low risk.

### `HARD12-020` — Home's page-local machine pause state desyncs from Play's pause; resuming from Home leaves playback muted

**Issue:** Home tracked the machine pause state in a `useState` that always started "running" on mount (HomePage remounts each tab visit) and was never told of pauses done elsewhere, so pausing on Play then going Home showed "Pause" for an already-paused machine. Worse, Play's pause path mutes the SID mixer before pausing and unmutes on its own resume, so resuming from Home ran `machineResume` without the unmute — the C64 resumed with the mixer still muted, Play still showed "paused" with a frozen timeline, and the user heard nothing until they found Play's Resume. There is no device endpoint to read the paused state back, so the two pages could not converge by polling.

**Fix:** Machine pause/resume state was lifted into a shared module-level `machineExecutionStore` written by both Play's and Home's pause/resume (and reset/reboot/stop) paths, and Home's resume goes through a shared helper that unmutes when a SID pause-mute was recorded. The store is device-scoped and cleared on device switch/stop so device A's state cannot bleed onto device B (avoiding the HARD11-002 class). Code comments at `machineExecutionStore.ts:13/22/80`, `useHomeActions.ts:50`, and `usePlaybackController.ts:1265/1392/1420` document the shared store and the pause-mute publication.

### `HARD12-021` — SID SSL upload writes the duration at subsong-1's slot regardless of songnr

**Issue:** The `.ssl` sidecar the firmware consumes is a per-subsong array of 2-byte BCD mm:ss entries indexed by `(songNr-1)*2`, but `createSslPayload(durationMs)` built a single 2-byte entry — always subsong 1's slot — and uploaded it alongside `songnr=N`. For every launch with `songNr > 1` (HVSC per-subsong playlist entries, the Play subsong picker), the played subsong's SSL slot was zero (the firmware memsets the region) and the resolved duration sat in the wrong slot, so device-side songlength behaviour was correct only for subsong 1. App-side auto-advance masked most of it, but whatever the device did at SSL expiry diverged from intent for multi-subsong SIDs.

**Fix:** The SSL payload builder now emits `songNr` entries (zeros for unknown slots) with the resolved duration written at index `songNr-1`, filling other slots from the per-subsong HVSC md5 durations array when available, so the firmware reads the correct duration for the launched subsong. A code comment at `sidUtils.ts:241` records that the firmware reads up to 512 bytes from the SSL sidecar and the entry must land at the `songNr` slot. This is a payload-only change.

### `HARD12-022` — Every tab transition fully mounts adjacent pages, and E2E probe mode disables exactly that behavior

**Issue:** During every tab transition the 3-panel runway mounted the adjacent pages, so a Home→Disks swipe transiently mounted `PlayFilesPage` and ran its full initialization (playlist repository hydration, sessionStorage session-restore reads, persistence/effect churn) before unmounting back to a placeholder. These transient instances are the documented root of BUG-040 and the hazard class behind several session-scoped-state findings, and they waste native/repo/storage work on low-end keypad hardware. Additionally, `renderPlaceholderOnly` is forced on when test probes are enabled, so E2E/probe runs never see transient mounts and the whole transient-instance bug class is structurally invisible to the deterministic harness.

**Fix:** This was deliberately not code-fixed. A code comment at `SwipeNavigationLayer.tsx:467` records that the mid-transition mount is real and load-bearing (the swipe preview renders actual page content, so replacing it with a placeholder is a visible-UX change, not a pure bugfix), so it was deferred as a separate, measured perf/UX task. The shipped mitigation is testability — a `SwipeNavigationLayer.test.tsx` HARD12-022 mount/unmount test — plus per-effect defenses in the pages that hold state across the overlap (BUG-040 wake lock, HARD12-006 volume session, HARD12-020 machine-execution state).

### HARD13

### `HARD13-001` — Capability-downgrade/close release-all is inert once the tier has already downgraded — held inputs stranded on the device

**Issue:** `releaseAll` cleared local held-set state but gated the on-device `release_all` on `tierRef.current === "full"`, and `tierRef` is updated to the new tier on every render before the `[tier]` downgrade effect fires. So on a full→kernal-fallback downgrade (a transient probe failure on the still-reachable u64, a connection blip forcing the conservative default tier, or a device switch) the guard was already false and no `release_all` was ever sent, leaving the direction or fire physically held on the device while the UI showed nothing held. The unmount cleanup effect had the same `&& tierRef.current === "full"` gate, and the existing tier-drop test only asserted the local held-set was empty, so the inert send went unobserved.

**Fix:** The release now fires whenever inputs were actually relayed, independent of the now-current tier: `lastSentHeldSetRef.size > 0` is only ever populated by the full-tier relay path, so it is a reliable "we owe the device a release" signal. `releaseAll` now sends when `hadRelayedInputs || tierRef.current === "full"`, and the unmount effect sends when `lastSentHeldSetRef.size > 0`. The tier-drop test was strengthened to assert `release_all` actually reaches the device (fail-before/pass-after proven); a residual device-switch case (where `getC64API()` already points at the new host) is documented, and the E1 follow-up (`e8552e28`) later added a pre-switch release choke point.

### `HARD13-002` — A coalesced+throttled input send can dispatch after an immediate release-all/port-swap, re-asserting a released input (stuck held input)

**Issue:** Ordinary coalesced sends wait out the device-safeguard throttle (`waitForMachineInputThrottle`, up to 200ms on CONSERVATIVE) before dispatching, while an immediate `release_all`/port-swap release bypasses the throttle. With a direction held and changing, the coalesced press sat behind the throttle; if the user then closed the sheet, hit panic, or swapped port, the immediate `release_all` fired and cleared the device, then the throttle resolved and the stale press dispatched after the release, re-asserting the input on the device (e.g. joystick "right" stuck on) with no local state to reflect it. It was recoverable only by re-opening the sheet and re-pressing, and the existing chaos test never advanced time past the throttle so the trailing dispatch was never observed.

**Fix:** A `sendGenerationRef` epoch was introduced in `sendEventsNow`: every immediate send bumps the generation, and each coalesced dispatch captures the generation at schedule time and drops itself if a later immediate send has superseded it. This closes both the release-all race and the identical port-swap race by discarding any throttled coalesced send that a subsequent immediate send has invalidated. A new chaos test drains the throttle after a panic release and asserts both that no send lands after the `release_all` and that the reconstructed device state is fully released (fail-before/pass-after proven); the later HARD18-001 change (`2bc5dbf2`) additionally bumps the generation on batch-failure recovery.

### HARD15

### `HARD15-001` — Kernal-fallback typing races: concurrent unserialized keyboard-buffer injections garble input and burst-load the c64u

**Issue:** On the kernal-fallback tier (every current c64u on firmware 1.1.0, where `GET /v1/machine:input` returns 404), every keystroke on the Keys tab fired `void injectAutostart(...)` with no serialization between calls. `injectAutostart` is an async poll-then-write loop (read `$C6` until 0, write `$0277`, write `$00C6`, delay 120 ms) taking ≥300 ms per keystroke, so typing at normal cadence ran two or more injection loops concurrently — interleavings dropped keystrokes or wrote a mix of two payloads to `$0277`, garbling input, while N concurrent poll loops multiplied `machine:readmem` traffic onto the wedge-prone c64u. Typing `LOAD`/`RUN`/BASIC commands — the primary Type-mode use case — was unreliable in a way that looked like device flakiness.

**Fix:** Commit `2ea35c1f` serialized all kernal-fallback injections through a FIFO promise queue (new `kernalFallbackInjector.ts`, mirroring `machineInputThrottle`'s pattern) so concurrent poll-then-write loops can no longer race on the shared C64 keyboard buffer or burst-load the device, with each failed injection rejecting only its own caller rather than poisoning the queue. A code comment at `kernalFallbackInjector.ts:20` records the serialization. (This wiring was briefly reverted by a concurrent commit and restored in `fcda2549`.) This eliminates both the interleaved-payload garbling and the redundant concurrent poll traffic against the device.

### `HARD15-002` — machine:input probe caches transient errors forever

**Issue:** `probeMachineInputCapability` cached every outcome — including `"error"` (network hiccup, 1500 ms timeout) and `"auth-required"` — in a module-level `Map` keyed by `deviceId|firmware|core`, with no TTL and no production invalidation path. A U64 owner opening the Remote Input sheet during a Wi-Fi roam got the probe timeout cached as `kernal-fallback`, so from then on every sheet open returned the cached failure instantly and disabled Joystick mode with a false "Joystick relay requires Ultimate firmware with machine:input support" hint until the app process restarted. The `auth-required` variant was analogous: a 403 cached before the user fixed the stored password stuck after the fix.

**Fix:** Commit `c6abff51` stopped caching non-definitive outcomes — only `available`/`missing`/`hardware-unavailable`/`unsupported-family` (real, stable firmware answers) are cached, while `error`/`auth-required` results are returned without insertion, so the next sheet open re-probes naturally. Definitive statuses still cache, keeping the c64u (404) sheet-open hot path request-free. A code comment at `capabilityModel.ts:255` records caching only definitive firmware answers.

### `HARD15-004` — Owed-release signal derived from the wrong state in `setPort` and the tier-downgrade effect

**Issue:** HARD13-001 established that "we owe the device a release" must be signalled by `lastSentHeldSetRef` (what was actually relayed), never by tier or local UI state, but two sites still used the wrong signal. `setPort` gated its old-port release on `pendingHeldSetRef.size > 0 && tier === "full"`, so releasing the stick (pending→empty, release flush scheduled for +40 ms) then tapping the port switch within that window skipped the release and wiped `lastSent` to empty, leaving the direction physically pressed on the old port with no local state reflecting it. The tier-downgrade effect gated `releaseAll` on local `heldJoystickInputs`, and `flush` no-oped-and-wiped `lastSent` on a downgraded tier, orphaning a relayed press.

**Fix:** Commit `fcda2549` (originally `bd15e3ea`) changed `setPort` and the tier-downgrade effect to gate the owed-release on `lastSentHeldSetRef` (what was actually relayed), and stopped `flush` from overwriting `lastSent` on a downgraded-tier no-op. Code comments at `useRemoteInputSession.ts:271/422/597` record gating on `lastSentHeldSetRef` rather than pending/local UI state. The fix had been accidentally reverted by a concurrent chaos-regressions commit and was restored in `fcda2549`.

### `HARD15-005` — Movement-style switch while a direction is held strands the direction pressed on the device

**Issue:** The Analog/D-Pad/Swipe style selector stayed enabled during play; holding a direction with one thumb and tapping a different movement style unmounted the outgoing control, whose release path was exclusively pointer-event-driven (`onPointerUp`/`onPointerCancel`), which an unmounted element never receives. There was no unmount or style-change cleanup stripping directions from the session held set, so the direction stayed in `heldJoystickInputs` and pressed on the device indefinitely — an unbounded stuck-input window, the highest-value class the program tracks. The SwipePad's unmount cleared its auto-release timer but likewise left the swipe direction held.

**Fix:** Commit `fcda2549` (originally `bd15e3ea`) made a movement-style switch strip the held directions (while preserving fire) and reset the outgoing control's local gesture state, so no direction survives the unmount. A code comment at `VirtualJoystick.tsx:137` records that switching movement style unmounts the outgoing control before its pointer-release can fire, so the directions must be stripped explicitly. The strip is scoped to direction inputs so a concurrently-held fire button is preserved.

### `HARD15-006` — Transient connection blip bounces the sheet from Joystick to Keys mode and never bounces back

**Issue:** While the sheet was open on a full-tier U64, any transient `isConnected` flap made `useRemoteInputCapabilityTier` synchronously reset to `{tier: "kernal-fallback", loading: false}`, and the sheet's smart-default effect — guarded only by `probeStartedRef`, still `true` from the earlier successful probe — then yanked the user out of Joystick into the Keys tab mid-game. When the connection returned a second later and the tier re-resolved to `full`, nothing switched the mode back, so the user had to notice and manually re-tap Joystick; in immersive/game mode it additionally force-exited immersive, compounding the disruption. The user had no indication the mode swap even happened beyond suddenly finding themselves on the wrong tab mid-session.

**Fix:** Commit `c6abff51` added a `resolved` flag to the capability tier's state, true only for a genuine definitive probe answer and false for both the default/reset shape and non-definitive results, and gated the sheet's smart-default effect on `resolved` instead of the deleted `probeStartedRef`. A connection blip now just disables the Joystick tab with a hint through the blip and re-enables it on reconnect, instead of silently swapping to Keys and never swapping back. Code comments at `useRemoteInputCapabilityTier.ts:21/26` and `RemoteInputSheet.tsx:163` record gating on `resolved`; the genuine kernal-fallback bounce on a real c64u is preserved.

### `HARD15-007` — Send-failure recovery assumes the failed batch was not applied

**Issue:** When a `sendMachineInputBatch` POST failed, the handler reset both `lastSentHeldSetRef` and `pendingHeldSetRef` to empty, encoding the assumption that the device did not apply the batch. That holds for connection-refused and HTTP 400 (whole-batch validation), but not for a timeout, where the press may have landed and the response been lost (a documented Wi-Fi-roam failure). After the reset, the user releasing the stick produced `pending = empty` and the next flush diffed empty→empty = no events, so the release was never transmitted — the direction/fire stayed physically pressed while the UI showed everything released, recoverable only by Release All or sheet close.

**Fix:** Commit `fcda2549` (originally `bd15e3ea`) made the full-tier error handler fire one recovery `release_all` when a release could be owed, guarded against looping on its own failure, since `release_all` is idempotent and self-heals both the applied and not-applied cases. A code comment at `useRemoteInputSession.ts:197` records that a batch failure (a timeout especially) does not guarantee the batch was not applied. The commit also fixed a test-pollution bug where a persistent `mockRejectedValue` (instead of `mockRejectedValueOnce`) broke later tests in the same file.

### HARD16

### `HARD16-001` — Startup saved-device sweep stamps the reachable device's identity onto the still-selected unreachable device

**Issue:** On startup with the selected device A powered off and saved device B on, the discovery window expired and `tryReachableSavedDeviceFallback` called `verifyCurrentConnectionTarget` for B while A was still selected. `setSnapshot` fed B's `/v1/info` `deviceInfo` to `rememberSelectedSavedDeviceIdentity`, and the HARD12-011 guard did not fire (its probe window is only opened by `executeSavedDeviceSwitch`, and this was the flow's own synchronous result), so `completeSavedDeviceVerification` stamped device B's product family, firmware, hostname, and unique_id onto device A's saved record and marked it verified "now" for a powered-off device. This corrupted the identity that drives per-device safety presets and product-gated config writes, and the hand-rolled switch also skipped `setStoredFtpPort`/`setStoredTelnetPort`, so B got wrong-port FTP/Telnet until a manual re-switch.

**Fix:** Commit `9721e4c1` (Batch A) made `tryReachableSavedDeviceFallback` select the reachable candidate and apply its FTP/Telnet ports before verifying, inside the HARD12-011 probe window, mirroring `executeSavedDeviceSwitch`. Verification then stamps identity onto the correct (now-selected) device, and on verification failure the selection stays on the candidate unverified (switch-path failure semantics, no rollback). A code comment at `connectionManager.ts:842` records selecting and applying the reachable device's identity/ports before verifying.

### `HARD16-002` — "resume" recovery policy is dead code

**Issue:** The connection manager had a fully built "resume" policy — on a resume-triggered discovery it sweeps the other saved devices and can fall back to LAN discovery — but nothing ever invoked it: there was no Capacitor `appStateChange`/`resume` listener, and `ConnectionController`'s `visibilitychange` handler only re-armed the OFFLINE background timer, whose probe targets only the selected device. So foregrounding a long-lived app process with the selected device off left the app probing the dead device forever and stuck `OFFLINE_NO_DEMO` while a reachable saved device sat unused, contradicting the cold-start behaviour of the same build. Because Android keeps app processes alive for long periods, the resume path is arguably more common than cold start.

**Fix:** Commit `9721e4c1` (Batch A) made `ConnectionController` run `discoverConnection("resume")` on a hidden→visible transition when the app was hidden ≥ 30s, the state is `OFFLINE_NO_DEMO`, background rediscovery is allowed, and no discovery results await selection; short blur/focus flips still only re-arm the selected-device probe. This was later further extended by HARD18-007 (manual/background reconnect escalating to the sweep + LAN discovery). Code comments at `ConnectionController.tsx:48/60/227` record the long-hidden resume escalation. HARD16-001 landed first so the resume sweep inherits the fixed identity-stamping path.

### `HARD16-003` — Cursor hold-repeat outruns the serialized kernal-fallback injector

**Issue:** The new cursor hold-repeat (`useHoldRepeat` fires every 100 ms after a 400 ms delay) unconditionally enqueued a full `injectAutostart` cycle per repeat into the HARD15-001 FIFO, but each injection costs ~600+ ms serially on the c64u's CONSERVATIVE profile against a 10/sec producer. A 3-second hold left ~25+ queued injections that kept moving the C64's cursor for 15+ seconds after release, uncancellable (nothing drains the queue — Release All and sheet close do not), while the backlog sustained ~5 `machine:readmem`/`writemem` calls/sec against the single-threaded, wedge-prone firmware. This coupled the program's two top protected properties — runaway uncancellable input and sustained wedge-class REST load — on the flagship device.

**Fix:** Commit `4f0ce5ee` (Batch B) gave `kernalFallbackInjector` a `dropIfBusy` option and a `pendingCount` counter so a cursor repeat is dropped once one injection is in flight and one is queued (`pendingCount > 1`), and `sendCursor`'s fallback branch passes `dropIfBusy: true`; a held cursor key now stops ~one injection after release. `sendChar`/`sendKeyboardInputs` still queue every typed character so typing is never dropped. Code comments at `useRemoteInputSession.ts:554` and `kernalFallbackInjector.ts:31` record the drop-while-busy bound, and a held-then-released cursor still emits at least the initial press.

### `HARD16-004` — Enrichment-namespace flip migrates the previous identity's cached items into the new namespace

### `HARD16-005` — Model-absent config items re-interrogated on every Home/Disks mount

### `HARD18-001` — Queued coalesced input batch re-asserts a press after failure-recovery release-all, stranding a held input on the device

**Issue:** When a user played with the remote joystick over flaky WiFi, an in-flight machine:input batch (e.g. "press RIGHT") could hang for seconds on a native read timeout while later flushes (autofire ticks, further interaction) queued behind it in the serialized FIFO. When the hung batch finally failed, the catch handler reset all held-set refs to EMPTY and fired a recovery release-all, but it never bumped `sendGenerationRef`, so the already-queued successor batch — computed against the now-abandoned model — still passed the generation check, dispatched, and re-asserted "press FIRE" (or a direction) on the device. Because local `lastSentHeldSetRef` was EMPTY, the user lifting their finger produced an empty→empty diff that sent nothing, leaving the input physically stuck on the real machine with the UI showing nothing held, and defeating the very HARD15-007 recovery mechanism built for this flaky-network case.

**Fix:** Fixed in commit 2bc5dbf2 by bumping `sendGenerationRef.current += 1` in the `sendEventsNow` catch handler alongside the existing ref resets, so every queued-but-not-yet-dispatched coalesced batch drops itself via the generation check. This reuses the same supersede doctrine HARD13-002 already established for immediate sends, meaning any stale batch computed against the abandoned model is discarded and the next genuine user change re-syncs from a clean slate. The change is a single line consistent with existing patterns, with the risk limited to dropping stale batches that would otherwise re-assert an abandoned input.

### `HARD18-002` — SHIFT LOCK indicator survives releaseAll while the device's left_shift is released, so the UI lies and typing goes out unshifted

**Issue:** On the Type tab (full tier), a user could tap SHIFT LOCK so the device held `left_shift` and the lock key lit up, then lock the screen, switch apps, or press the panic (release-all) button. `releaseAll` cleared the session's held keyboard set and genuinely released `left_shift` on the C64, but `useKeyboardHoldDispatch`'s component-local `shiftLocked` state was never reset by that path. On resume the SHIFT LOCK key still rendered as locked while every subsequent keypress relayed without shift, so the user saw the lock engaged yet got unshifted characters/graphics on the machine, with recovery only possible by guessing to toggle the (wire-no-op) lock off.

**Fix:** Fixed in commit e3ab13c8 by having the consumer lock state reset when `releaseAll` clears the held set: an effect drops `shiftLocked` when the held keyboard inputs no longer contain `left_shift` while the lock is engaged and the modifier is not physically held. This heals the consumer-side latch that the transport-only `releaseAll` previously left stale, keeping the SHIFT LOCK indicator honest across backgrounding and panic-button use. The fix is scoped to lock-state bookkeeping and guarded so it never releases the lock during normal typing.

### `HARD18-003` — Health-check CONFIG pulse revert reuses the run's aborted signal, durably leaving LED intensity / mixer volume changed on the device

**Issue:** The Diagnostics health check verifies config write round-trips by pulsing a real user-facing value (LED "Strip Intensity" ±16 of a 0–31 range, or "Vol UltiSid 1"/"Vol Drive 1" ±16 of a -64..0 range). When the pulse write succeeded but the run's AbortSignal then fired (12s global deadline on a slow device, a superseding run, or the user leaving diagnostics), the `finally` block attempted the revert with the same, now-aborted signal; because `createTimedRequestSignal` pre-aborts the request controller for an already-aborted outer signal, the revert never reached the wire and the failure was swallowed into a warn log. The result was that a diagnostic — advertised as effectively read-only and the recovery tool for a degraded device — visibly and audibly changed the user's LED brightness or SID/drive volume by half the range and left it that way, exactly on the slow devices where the deadline abort is most likely.

**Fix:** Fixed together with HARD18-008 in commit 76be904d by running the CONFIG-pulse revert with a detached AbortController and its own short timeout instead of the run signal, so the restoring PUT still reaches the wire after a run abort. The revert targets the exact previously-read value, and at worst issues one extra small PUT after abort, so the diagnostic no longer strands persistent audio/LED settings. This detached-signal revert also underpins the HARD18-008 switcher-abort scenario, which shared the same root cause.

### `HARD18-004` — Sheet close clears physical-key tracking but not its last-contributed input set, wrongly releasing a later touch-held input

**Issue:** On a keypad-class device the user could hold a physical D-pad direction (RIGHT) in Joystick mode and close the Remote Input sheet (Android back / X) while the key was still down; `handleOpenChange` cleared `heldPhysicalKeysRef` and called `releaseAll`, but left `previousPhysicalInputsRef` holding `{RIGHT}`. Because the sheet component stays mounted for the life of the page, that stale ref survived, so when the user later reopened the sheet, touch-held RIGHT via the joystick, then pressed physical FIRE, `recomputePhysicalHeldSet` merged current physical `{FIRE}` against stale previous `{RIGHT}` and deleted RIGHT from the merged set even though a live touch hold owned it. The device released RIGHT mid-gesture — the character stopped moving with the user's thumb still on the stick — until the next pointer move re-asserted it, an input-fidelity gap in combined physical+touch play.

**Fix:** Fixed in commit fc0a7e43 by adding `previousPhysicalInputsRef.current.clear()` next to the existing `heldPhysicalKeysRef.current.clear()` in `handleOpenChange`, mirroring the output-mode-change effect that already clears both refs for exactly this failure class. Clearing the stale contribution record on sheet close means a subsequent touch-held input is no longer wrongly stripped by a phantom physical key from a prior session. The change is negligible risk since it only clears a ref the sibling mode-change path already clears.

### `HARD18-005` — Remote input relay funneled through the shared single-slot REST scheduler freezes joystick/keyboard relay behind any slow REST call

**Issue:** Remote input during playback is the marquee combined flow, but every machine:input batch was pushed through `withRestInteraction` into `restScheduler`, whose hard `REST_MAX_CONCURRENCY = 1` gave no exemption to the input lane. Whenever any other REST call was in flight — a playlist auto-advance mount (up to 8s), a SID/PRG upload (5s), a RAM block write (15s), or a degraded-device poll riding out its 3s timeout — the user's joystick presses and releases stalled for seconds behind the running slot-holder and then landed late in a burst. The design contract in the code explicitly says the relay must NOT be governed by the shared concurrency semaphore, but the implementation only swapped the native serialization semaphore while still routing through the single-slot scheduler, so delayed releases read as a stuck joystick and the FIFO's queued stale batches later dispatched back-to-back into the fragile firmware; tests missed it because `withRestInteraction` short-circuits under `isTestEnv()`.

**Fix:** Fixed in commit 924996a7 by exempting input-lane traffic from the shared `restScheduler` so it runs off the single bulk slot, relying on `serializeMachineInputRequest` plus `machineInputThrottle` for non-overlap — the documented safety model for that endpoint. This keeps input throughput at its intended floor independent of polling/mount/upload load, so a held direction's release is no longer delayed behind bulk REST work. The native serialize split already permits one bulk plus one input connection to overlap, so the wedge-safety story does not regress beyond the documented HARD13 design.

### `HARD18-006` — Write-path backoff/cooldown sleeps hold the exclusive REST slot, freezing all REST traffic including forced health-check probes

**Issue:** When a device was flaky and the REST error streak had pushed `restBackoffUntilMs` up to ~6s into the future, a user tapping a config slider or machine action entered the scheduler slot and slept out the remaining backoff (and up to 1.2s cooldown) inside the only slot, since only reads deferred their waits via `getReadyAtMs` while non-GET writes slept in-slot. A subsequent "Run health check" — whose forced probe carries every bypass flag and is documented to "MUST always reach the wire" — then sat queued behind the sleeping write because a queued task cannot preempt the running slot-holder. The whole REST surface (drive status, config reads, playback controls, the recovery-tool health check) froze for seconds precisely during degradation, and on a slow device the delayed probe ate half the 12s deadline, pushing runs into the abort path that HARD18-003 showed leaves a pulsed config value un-reverted.

**Fix:** Fixed in commit 52684da5 by giving writes the same `getReadyAtMs`-based deferral that reads already use, computing ready-at from `restBackoffUntilMs`/cooldown and letting the scheduler's deferred-drain timer hold them out of the slot, deleting the in-slot sleeps. Backoff and cooldown pacing is preserved but no longer occupies the exclusive slot, so a forced diagnostics probe can reach the wire without waiting out another request's sleep. Write ordering relative to other writes stays stable through the same single-slot serialization, and the drain timer wakes correctly when backoff shortens.

### `HARD18-007` — Manual reconnect and background rediscovery never escalate to the saved-device sweep or LAN scan, stranding the app offline after a device IP change

**Issue:** When a c64u wedged, was power-cycled, and got a new DHCP address, the app was left probing the stale stored host: tapping the Offline badge ran exactly one `probeOnce` against the stored host and transitioned to OFFLINE_NO_DEMO, and the background rediscovery loop re-probed the same stale host forever. **Fix:** Fixed in commit 9597654c by allowing the manual (and escalated background) reconnect path to fall through to the saved-device sweep and LAN discovery after a failed direct probe, instead of transitioning straight to offline. A device that changed IP mid-session is now found and reconnected without an app restart. Manual-tap coalescing is retained and the LAN scan is rate-limited so escalation does not produce a scan burst.

### `HARD18-008` — Device Switcher picker re-runs the CONFIG write pulse against every saved device every 10s, and closing it mid-pulse leaves LED/volume changed on multiple devices

### `HARD18-009` — Stop during an in-flight track transition is silently overridden, letting the queued playItem re-assert isPlaying and re-arm auto-advance

**Issue:** During a playlist auto-advance (or a tapped row/Next), `playItem` could be mid-`executePlayPlan` — a multi-second reboot+mount+run window for disk items — when the user pressed Stop. `handleStop` was not routed through the play-transition queue and set no supersede marker: it issued its reset immediately (racing the in-flight mount/run on the wire) and cleared `isPlaying`, the auto-advance guard, duration, and the clock, but the queued `playItem` continued obliviously and, on resolving, unconditionally set `isPlaying(true)`, re-armed `autoAdvanceGuardRef`, and restored duration. **Fix:** Fixed in commit b10139d0 (M5) by introducing a play-generation counter: `handleStop` bumps it (and still runs immediately, unqueued), and `playItem` captures the generation at entry and skips all post-`executePlayPlan` state writes — issuing a follow-up reset if the launch already happened — when it detects it was superseded. ### `HARD18-010` — "Load from flash" / "Reset to default" never invalidate per-item config queries, leaving Home/Disks/Play controls showing pre-action values

### `HARD18-011` — Saved-device switch while Play is unmounted orphans the background-execution session, leaving wake lock/foreground service running with no owner

### `HARD18-012` — Telnet action execution runs concurrently with REST polling, and power-cycle's outage trips the circuit breaker into a false-offline right after the user's own action

### `HARD18-014` — REU "Preload on Startup" is configured against volatile `/Temp`, so the preload can never fire and the success toast is false

**Issue:** When a user tapped "Preload on Startup" for an REU snapshot, the app uploaded the REU image to `/Temp/<name>.reu`, telnet-navigated to that file, and selected the firmware's "Preload on Startup" action, which stored the `/Temp/...` path in the device's REU preload config, then toasted "REU preload configured". ### `HARD18-015` — Saving a snapshot while the machine is user-paused silently resumes it and clears pauseMutePending, stranding the SID mixer muted

**Issue:** When a user paused playback from Play (which mutes all enabled SID volumes on the device and sets machine state "paused" with `pauseMutePending = true`) and then tapped "Save Snapshot" on Home to capture the frozen moment, `dumpRamRanges` → `runPaused` unconditionally called `machinePause()` then `machineResume()`, resuming the machine and destroying the user's pause so the game/demo kept running. `handleSaveRam` then unconditionally called `setMachineExecutionState("running")`, which cleared `pauseMutePending`. The consequence chain was three user-visible wrongs from one tap: the deliberate pause was undone, the machine ran with the SID mixer still muted (its only unmute triggers now gone), and the Home Pause/Resume button read "running" — inverted from what the user did — with the muted mixer reading as "my Ultimate's sound broke"; the same applied to CPU+RAM save and to a snapshot restore performed while paused.

### `HARD18-016` — Developer-only feature-flag overrides stay applied after developer mode is exited, leaving hidden features enabled or background execution disabled with no editable control

**Issue:** A user could enable developer mode, flip a developer-only flag — e.g. enable `lighting_studio_enabled` (a variant-excluded experimental device-touching feature) or disable `background_execution_enabled` — then turn developer mode off; `applyDeveloperMode(false)` only recomputed visibility/editability and never cleared the persisted overrides, while `computeResolution` kept applying them to the resolved value. This keeps overrides available for the next developer session while removing the invisible, uneditable divergence in normal use.

### `HARD18-017` — Deleting a mounted disk ejects via a bare unmountDrive and deletes the mount override, showing a ghost mount for up to a full poll interval

**Issue:** When a user deleted a disk that was mounted in a drive, `handleDeleteDisk` ejected it with a direct `api.unmountDrive(drive)` — bypassing `runDriveMutationWithSettledPolling`, so there was no polling pause, no `["c64-drives"]` invalidation, and no settle wait — and then deleted the `mountedByDrive[drive]` key instead of setting the forced-empty `""` marker that `handleEject` uses. **Fix:** Fixed in commit 7f3b54df by routing the delete-time eject through the established `runDriveMutationWithSettledPolling` wrapper and setting the forced-empty `""` mount override (timestamped), reusing `handleEject`'s validated body for each mounted drive before removing the disk from the library. Because it replaces an ad-hoc eject with the already hardware-proven mutation wrapper, the change is low risk.

### `HARD18-018` — Pause/resume/Home volume restore ignore the firmware's "Vol Master" item, writing multi-item per-SID batches where a single master PUT would be safer and complete

**Issue:** On a U64 running firmware 3.15+ the Audio Mixer exposes `Vol Master`, a single item that mutes/scales all audio sources; the Play slider and manual mute already collapsed to `[Vol Master]` when present, but the machine-pause mute chain did not — `muteBeforeMachinePause` hardcoded `enabledSidVolumeItems` (up to 4 per-SID items), so pause, resume, failed-pause rollback, and Home's cross-page resume restore each wrote a multi-item Audio Mixer batch. **Fix:** Fixed in commit 06246ce1 by threading the master-aware item selection into the pause-mute path so, on master-capable firmware, mute collapses to the single `Vol Master` item and prefers its explicit `OFF` option for true full silence (falling back to the per-SID path, and to -42 dB, when `Vol Master` is absent). Resume restores the same item(s) via a persisted snapshot that records which item set it captured, keeping manual and pause mute semantics consistent and dropping N-item POST bursts to single writes on master-capable devices. Old per-SID persisted snapshots still restore correctly, and this was HIL-covered for Vol Master OFF mute/restore. This collapses the highest-severity write pattern on the hottest mute path down to a single PUT wherever the firmware supports it.

### `HARD18-019` — Lighting Studio reconciler writes full multi-item surface payloads via POST /v1/configs on every change and retries a failing write forever

**Issue:** With Lighting Studio enabled (a developer flag that HARD18-016 showed can persist invisibly), every Preview tap, profile activation, quiet-launch transition, circadian boundary, or connection-sentinel change re-resolved the lighting state and, on any difference, sent the entire surface payload (typically 3–7 LedStrip items including `Strip Intensity`, whose repeated interactive writes are hardware-verified to crash the c64u network stack) as one multi-item `POST /v1/configs`. Worse, on write failure the catch only logged and did not record the signature, so the effect re-fired the same POST on every render, and with circadian automation a `setInterval` forced a re-render every 60s — an endless once-per-minute stream of the most dangerous write pattern against a degraded device. **Fix:** Fixed in commit 38c2d280 by making lighting writes delta-only (diffing next vs current state per item instead of always shipping the full surface), adding a failed-write backoff that records the failed signature with a next-attempt timestamp instead of retrying every render/tick, and gating the reconciler on the active device identity so a saved-device switch requires an explicit re-apply. This removes the unbounded retry storm and the auto-apply-on-switch, and shrinks each write to the minimum item set. Delta ordering follows the existing `orderFirmwareSafeConfigUpdates` doctrine so mode-dependent items are not missed.

### `HARD18-020` — Lighting Studio device preview misrepresents physical light behavior: no case/keyboard mixing, and "Off"/zero intensity still glows

**Issue:** The Lighting Studio preview had two mismatches against a physical C64 Ultimate. ### `HARD18-021` — Health-check CONFIG pulse read→write→revert races concurrent user Audio Mixer writes, reverting them to the pre-pulse value

**Issue:** On a device where the LED-strip pulse targets were unavailable, the CONFIG probe pulsed `Vol UltiSid 1` (or `Vol Drive 1`) by ±16 of a 64-step range as a read→write→delay→read-back→revert sequence spanning several REST round-trips. Any user mixer write landing inside that window — pause-mute (-42 dB), resume-unmute (0 dB), a volume commit, or Home's cross-page resume restore — was clobbered by the pulse's final revert to the captured pre-pulse value, e.g. a Resume during a manual health check writing 0 dB then getting reverted to -42 dB, leaving one SID muted while the UI showed unmuted with no error anywhere. **Fix:** Fixed in commit 1579659f so the CONFIG health-check pulse no longer clobbers concurrent mixer writes: it skips Audio Mixer targets while playback is active or a playback write burst is in progress, and re-reads the item before reverting, skipping the revert when the current value no longer equals the pulse's temp value (meaning another writer changed it). This leaves a concurrent user or reconciler write intact instead of silently overwriting it. The cost is one extra read per CONFIG probe plus reduced pulse coverage only while music plays.

### `HARD18-022` — Home's machine-reset actions leave the Play playback session armed, so auto-advance later reboots/mounts/runs the next track on the cleared machine

**Issue:** When a user played a playlist (auto-advance armed, native watchdog due-time set so playback survives navigation and screen-off), navigated to Home, and pressed "Reboot", "Reboot (Clr Mem)", or "Power cycle" — intending to stop or reset the machine — the C64 reset and the music died, but nothing told the playback session: `isPlaying` stayed true (UI showing "Playing" with a running clock over a silent machine) and the auto-advance guard and native due-time stayed armed. **Fix:** Fixed together with HARD18-023 in commit 6e9c65b9 (M3) by publishing a machine-takeover event from the reset-class Home handlers that the playback session subscribes to: on a user-initiated reset the session stops in place (cancels auto-advance, clears the native due-time, sets stopped, keeps playlist position) and shows a "machine was reset" notice. The same stop is applied to Play-page-unmounted sessions via the module-level background-execution manager, mirroring HARD18-011. The event is scoped to user-initiated Home/quick-action resets so playItem's own reboot-before-mount does not trigger the listener.

### `HARD18-023` — CommoServe "Run / Mount & run" launches content via direct executePlayPlan, leaving an armed Play session that auto-advance later reboots the machine mid-game into

**Issue:** When a SID playlist was playing (auto-advance armed, native watchdog set) and the user opened Settings → CommoServe archive and tapped "Run" / "Mount & run", `executeArchiveEntry` built a play plan and ran it directly against the device (reboot+mount+run for a disk entry) without any playback-session interaction. ### `HARD18-024` — REU snapshot save downloads the /Temp file on first observed change with no stability check, persisting truncated snapshots with full success UX

**Issue:** When a user tapped "Save REU" on Home, the firmware wrote the REU image into `/Temp` incrementally over many seconds (the UI itself warns "around 30 seconds"), but `waitForTempReuFile` polled the FTP listing every second and returned as soon as the target file first appeared or first differed from the pre-save listing — typically while the firmware was still writing. The workflow immediately FTP-downloaded only the bytes written so far, persisted the truncated image with its truncated byte count, and reported "REU snapshot saved" with the standard success toast, so the user's GEOS/RAM-disk contents were silently not saved. The loss surfaced only on a later restore of the corrupt/short image, potentially long after the original data was unrecoverable; larger REU sizes (4–16 MB, the ones users most want to snapshot) had the widest race window, and the config-snapshot save shared the pattern with a smaller window.

**Fix:** Fixed in commit 3c614687 by making `waitForTempReuFile` (and the config-file equivalent) keep polling after first detecting a candidate until its size and mtime are identical across two consecutive polls (quiescence) before returning and downloading. A post-download sanity check rejects byte lengths that are not a plausible REU size, surfacing a clear error instead of persisting a truncated image. This was HIL-covered against the `/Temp` write behavior, at the cost of at least one extra confirming poll (~1s) per save and a modest timeout raise for the largest images.

### `HARD18-025` — Writable upload-mounts silently discard everything the C64 writes, so in-game saves to library/archive disks evaporate on the next mount

**Issue:** When a user mounted a D64 from their local library or a CommoServe/archive download, the app uploaded the image bytes and the device mounted its own read-write copy (readwrite chosen by HARD9-012 so in-game saves don't fail with WRITE PROTECT ON). ### `HARD18-026` — Every telnet session outside useTelnetActions runs concurrently with REST polling, including once per track transition during playlists with config references

**Issue:** The polling-pause doctrine — a telnet menu session must not overlap REST connections on the single-threaded, wedge-prone firmware network task — was implemented in exactly one place, useTelnetActions' capability discovery. ### `HARD18-027` — Diagnostics "Retry connection" hard-codes port 80 and appends it to a host that may already carry a custom port, producing a malformed host:port:80 URL

**Issue:** For a saved device configured with a non-default HTTP port (port-forwarding / reverse-proxy setups), `getConfiguredHost()` returned e.g. `192.168.1.5:8080`, but the Diagnostics "Retry connection" handler called `validateTarget(host, 80)` with a hard-coded 80, and `validateTarget` unconditionally appended `:80` to produce `http://192.168.1.5:8080:80`. **Fix:** Fixed in commit ef2e0f37 by deriving the port from the stored device instead of hard-coding 80 in the retry handler, and hardening `validateTarget` to split any embedded port from the host before appending so a stored `host:port` string composes to a valid base URL (embedded port wins). The change is negligible risk and covered by the sibling switch-device path that already passed a real port.

### `HARD18-028` — Native HVSC updates never remove deleted/moved songs from the JS browse index, accumulating dead and duplicate entries that fail on play

**Issue:** On Android (the production ingestion mode "native"), applying an HVSC update — which routinely moves and deletes SIDs — had the native plugin extract, apply deletions to the on-disk library, and update its SQLite metadata DB, but the JS browse-index snapshot that actually serves the Play page's HVSC browsing and search was only ever added to. ### `HARD19-001` — Kernal-fallback tier: merged CRSR keys can never send cursor up/left

**Issue:** On the kernal-fallback input tier (Ultimate-II cartridges without `machine:input` REST support) the expanded on-screen keyboard renders no CursorPad — only two merged CRSR keys whose labels promise "hold Shift for up/left". The fallback `dispatch` handled the cursor case as `onCursor(action.direction)` with no check of the active SHIFT modifier, and then cleared the one-shot modifier latch on every ordinary key. So a user who tapped SHIFT (which lit up) and then a CRSR key moved the cursor the wrong way while the latch was silently consumed, making cursor up and cursor left completely unreachable on every tablet/desktop-web fallback-tier device.

**Fix:** The fallback dispatch cursor case now computes `shiftHeld = activeModifiers.has("left_shift") || shiftLocked` and dispatches `CURSOR_SHIFT_COMPLEMENT[action.direction]` when set, mirroring the full-tier handler; the one-shot latch is still consumed afterwards so SHIFT applies to exactly the next key. `sendCursor` already emitted a dedicated PETSCII byte per direction on the fallback tier, so the transport supported all four directions all along — only the UI never asked for up/left. ### `HARD19-002` — DirectionPadButton lacks KeyHoldButton's drag-off ref reset

**Issue:** `DirectionPadButton` armed a `handledByPointerRef` on pointer-down (to suppress the trailing synthetic click of a touch press) and cleared it only inside `onClick`. When a finger drags off the cell before releasing, pointer capture redirects the `pointerup` to the cell but the browser synthesizes no click for an off-element release, so the ref stayed `true`. The very next click-only activation of that cell — a keypad/focus-ring Enter/Space, assistive tech, or the E2E harness's synthetic click — was then silently swallowed, and both the Joystick D-pad and the Keys CursorPad render through this button.

**Fix:** Ported KeyHoldButton's already-proven `onPointerUp` geometric reset into `DirectionPadButton`: when the release lands outside the cell's bounding rectangle (and carries real, non-`(0,0)` coordinates, so a synthetic in-place release isn't mistaken for a drag-off), the ref is reset immediately. ### `HARD19-003` — VirtualDPad cells share joystick axes without contribution tracking

**Issue:** `VirtualDPad.setCellHeld(inputs, false)` unconditionally deleted every input of the released cell from the shared held-set. Because diagonal cells overlap cardinal cells on an axis (both the `up` cell and the `up-right` cell contribute `up`), a user holding `up` with one finger while a second finger grazed and released `up-right` had `up` deleted mid-game even though the first finger still physically held it. **Fix:** VirtualDPad now tracks which cells it currently holds in a ref and rebuilds its axis contribution from the union of still-pressed cells on each change, mirroring `RemoteInputSheet.recomputePhysicalHeldSet`: it removes only inputs this pad contributed last time and no longer holds, so directions held by other still-pressed cells (and inputs from other sources such as a touch-held FIRE) survive. An unmatched release for a cell that never registered a press is now a harmless no-op. Tests assert a second cell's release keeps the first finger's `up`, and that an unmatched slid-on release frees nothing.

### `HARD19-004` — One manual health check permanently pins the global health badge

**Issue:** After a manual Diagnostics health check recorded a verdict into `healthCheckState.latestResult`, `useHealthState` returned that pinned verdict for the app-wide badge whenever `latestResult` existed, with no staleness bound. So a check run during a transient device hiccup pinned a red Unhealthy/Degraded badge indefinitely even after hours of successful REST/FTP/Telnet traffic, and the reverse — a check run while healthy pinned Healthy through a later real degradation. The only production paths that cleared `latestResult` were a saved-device switch and another manual run, so the live trace-derived rollup became unreachable dead code.

**Fix:** Decision D1 implemented both directions. First, `transitionToRealConnected` in the connection manager clears a pinned Unhealthy/Unavailable `latestResult` on recovery, since reaching REAL_CONNECTED means the REST control plane just answered. Tests cover both the recovery and degradation directions plus the connection-manager clear.

### `HARD19-005` — Disk write-back has no device affinity

**Issue:** A materialized disk mount reused a deterministic per-drive work-file name on every device, but the pending write-back map recorded no device identity. If a user mounted a writable local disk on device A, switched saved devices to B, then ejected, the eject would FTP-read device B's stale work file for that drive and overwrite the user's local source file with it — silent local-file corruption of the P1 class. ### `HARD19-006` — Materialized disk write-back tracking is memory-only

**Issue:** The `materializedMounts` map that tracks pending write-backs lived only in memory, so Android killing the app process (a routine event under memory pressure) discarded it. A test imports the module fresh and asserts the rehydrated entry finalizes correctly.

### `HARD19-007` — Materialized mount loses drive-card identity

**Issue:** When a disk was materialized to a device work-dir path-mount, the on-device filename became the deterministic work-file name rather than the disk's own name. ### `HARD19-008` — Play-page disk launch drops a pending Home write-back

**Issue:** The Play page's `executePlayPlan` disk case did not thread the shared write-back dependencies through all of its mount paths, and its local-file path bypassed `mountDiskToDrive`. ### `HARD19-009` — Play timeline ignores an externally-applied pause

**Issue:** The Play page's auto-advance timeline was driven only by Play's own state, so a pause applied from another surface (a Home Pause of the machine) did not suspend Play's clock. A user who started a playlist, went Home, and paused would still have Play count down and, at the nominal track end, fire an auto-advance launch against a paused machine. The playback controller had no subscription to the shared machine-execution store that records external pause/resume.

**Fix:** The playback controller now subscribes to `machineExecutionStore`; an externally-applied pause suspends the played clock and clears the due-time, and a resume re-arms it. Play's own writes are routed through a guarded `writeMachineExecutionFromPlay` with a re-entrancy flag and a value-equality bail so the new subscription cannot cause a setState re-render loop (which would hang `test:coverage`), and a belt-and-suspenders machine-state gate guards `handleNext("auto")`. A harness test drives an external pause and asserts no launch fires and the state mirrors, with the full playFiles suite green and no coverage hang.

### `HARD19-010` — Home pause does not mute the SID mixer first

**Issue:** When the machine is paused from Home, the Play page's resume path expects the SID mixer to have been muted before the pause so it can restore the prior volumes — but Home's pause did not perform that mute-before-pause bracket. The result was that a Home pause could leave audio behaviour inconsistent with a Play pause, and the resume path had no persisted snapshot of the pre-pause mixer state to restore. Home and Play had diverged on the pause-mute contract.

**Fix:** Home pause now mutes the SID mixer first via a new `capturePauseMuteToPersistedSnapshot` that mirrors the restore shape and reuses Play's exact pure SID helper functions, writing a single Audio Mixer batch POST (`updateConfigBatch`, per the settled config-transport rule), then records `pauseMutePending` and persists the snapshot; a pause failure rolls the mute back. Rather than literally extract Play's `muteBeforeMachinePause` (too coupled to `useVolumeOverride` reactive state), an imperative sibling was built from the same pure functions so it cannot diverge from resume. Tests cover the capture helper and the mute-before-pause plus rollback behaviour.

### `HARD19-011` — Snapshot/REU restore doesn't publish the machine takeover

**Issue:** Restoring a RAM/CPU snapshot or loading into the REU changes the running machine state, but these actions did not publish the machine-takeover event that Play uses to stop auto-advancing over a machine the user just commandeered. So after a restore, an armed playlist could launch the next track over the freshly restored session — the same cross-page stale-state class that the HARD18-022 takeover contract exists to prevent. The restore paths simply missed the takeover publish.

**Fix:** The snapshot restore (RAM and CPU) and the REU load-into-REU success paths now publish the machine takeover via the shared `publishMachineInterrupt` helper, honouring the `endsPaused` flag so a restore that leaves the machine paused is reflected. Rather than add a new reason enum, the existing `home-reset` reason was reused. A `useHomeActions` test asserts the restore publishes the takeover.

### `HARD19-012` — Reachable-device fallback switch skips retarget hygiene

**Issue:** `tryReachableSavedDeviceFallback` in the connection manager is a second device-switch path that runs outside React, and it selected the new saved device without performing the hygiene the canonical `executeSavedDeviceSwitch` does (remote-input release, toast/health clearing, execution reset, background-execution stop, query invalidation, injection-queue drain). Because it ran in a lib context it also had no access to the React Query client needed to invalidate device-scoped caches, so the new device could render device A's cached payloads. The two switch paths had drifted apart on cleanup.

**Fix:** A shared `prepareForDeviceRetarget(fromDeviceId, toDeviceId)` helper was extracted with the hygiene block, plus a `queryClientRegistry` so lib-context code can reach the app's single React Query client (registered from App.tsx); the fallback path now calls the helper before `selectSavedDevice`. `executeSavedDeviceSwitch` was deliberately not rewired to the helper because its steps straddle a fallible password-resolve and API-retarget/verify boundary whose ordering encodes HARD12/HARD16 invariants; the helper mirrors it with a cross-reference comment. Tests cover the helper and assert the fallback calls it before selecting the device.

### `HARD19-013` — Kernal-fallback failed injection pins "Reconnecting…" forever

**Issue:** On the kernal-fallback tier, a single transient keyboard-buffer injection failure set `connectionStatus` to `"error"`, flipping the Remote Input sheet header to "Reconnecting…". But no fallback-tier code path ever set the status back — `setConnectionStatus("idle")` lived only in the full-tier `sendEventsNow` chain, while all four fallback dispatchers only ever set `"error"` in their catch. So after one WiFi blip the user typed successfully for the rest of the session under a permanent false "Reconnecting…" banner, nudging them toward disruptive recovery like restarting the app or power-cycling the device.

**Fix:** Each of the four kernal-fallback dispatchers (`sendChar`, `sendKeyboardInputs`, `sendCursor`, `sendSpecialKey`) now chains `.then(() => setConnectionStatus("idle"))` before its `.catch`, so a successful injection clears a prior transient error, mirroring the full-tier chain. This lets the indicator self-heal from success evidence that was previously discarded. A hook test fails one injection to reach `"error"`, then succeeds the next and asserts the status returns to `"idle"`.

### `HARD19-014` — Archive/CommoServe disk write-back persists into a 10-minute in-memory LRU

**Issue:** A CommoServe/archive disk materializes with an `archive-cache` write-back target, which counts as a "writable source", so the mount outcome was "materialized" and the one-time "changes won't be saved" advisory was suppressed. But its write-back only lands in an in-memory Map with a 10-minute TTL and a 100-entry LRU, so remounting the same disk later (or after restart, or after cache eviction) re-downloads the original image and the in-game saves are gone — silently, and without the warning a plain transient mount at least shows once. The "persisted" contract was more misleading than the transient path.

**Fix:** Per decision D2, archive-cache materialized mounts now show a one-time session-scoped advisory at mount time (a separate flag so it does not conflate with the never-saved transient advisory), and on eject the flow offers a durable "Save a local copy": `finalizeDiskWriteBack` returns an `archiveCopyOffer`, and `saveArchiveDiskCopyToLocalFolder` prompts for a folder and writes the cached bytes there. Cancelling the picker is silent; a lost cache or write failure is surfaced. Tests cover the offer for archive vs local-tree write-back, the save-copy saved/no-bytes/cancelled outcomes, and the eject action wiring.

### `HARD19-015` — Native SAF writeFileToTree delete-then-create can destroy the original file

**Issue:** The Android `FolderPickerPlugin.writeFileToTree` overwrote an existing document by deleting it and then creating a replacement. Any failure after the delete — `createDocument` returning null, `openOutputStream` failing, the write throwing, or process death — destroyed the user's original file outright, the most severe data-loss shape in the app, and it sat on the write-back path used to persist in-game disk saves. A transient SAF error at the wrong moment meant the original disk image was simply gone.

**Fix:** The plugin now overwrites in place by opening the existing document's URI with write-truncate mode (`openOutputStream(uri, "wt")`) instead of delete-then-create, and only creates a new document when the file is absent; disk images are fixed-size so truncation is a non-issue and any pre-write failure leaves the original present. Robolectric regression tests use the real (pure) `DocumentsContract` URI builders plus a mocked `ContentResolver` (static mocking of `android.*` is unavailable under Robolectric): the happy path asserts an in-place `"wt"` write to the existing document's URI and resolve, and the fault path asserts the write was aimed at the existing URI (not a created one) then rejects — proving a failure cannot destroy the original. Validated with `:app:testDebugUnitTest` under JDK 21.

### `HARD19-016` — Forced post-ingestion songlengths reload swallowed by load coalescing

**Issue:** `runLoad` in the songlengths service coalesced any concurrent request onto the in-flight load and returned, discarding the new request's `force` flag. So the post-ingestion `reloadHvscSonglengthsOnConfigChange({force: true})` — the fix for the historical install-gate bug — was swallowed whenever a cold-start or other load was already running: the awaited load ran un-forced and pre-commit (`installedVersion` still 0), discovered nothing, and every HVSC song stayed pinned at the default 3:00 duration until an app restart. The trigger is a plausible race between ingestion completion and a Play-page cold-start load.

**Fix:** `runLoad` still coalesces un-forced requests onto the active load, but a forced request now chains exactly one forced follow-up `loadInternal` after the active load settles (concurrent forced requests coalesce onto a single `pendingForcedReload`), so the forced discovery actually runs once the pre-commit load is done. This guarantees a `force: true` post-ingestion reload is never dropped. A test holds a cold-start load open, issues `force: true` mid-flight, and asserts `reloadOnConfigChange` is not called until the cold-start settles and then runs once.

### `HARD19-017` — Kernal-fallback injection queue survives a saved-device switch

**Issue:** The kernal-fallback injection queue held closures over the shared C64 API proxy with no record of which device each injection was enqueued for, and each injection is a multi-request poll/write loop spanning seconds. If a fallback-tier user typed or held a cursor key and then switched saved devices, the remaining queued injections kept draining against whatever device the singleton now pointed at, typing phantom PETSCII into device B's live keyboard buffer (and, in a torn variant, corrupting the queue count). Multi-device owners switching right after using Remote Input hit this routinely.

**Fix:** Injections now capture device affinity (host + connection epoch) at enqueue time and skip if the device changed before they run, plus a `shouldAbort` check mid-injection; a new `drainKernalFallbackInjectionQueue()` is wired into both switch paths (`prepareForDeviceRetarget` and `executeSavedDeviceSwitch`). This ensures a device switch cancels in-flight and queued injections rather than replaying them against the new device. Tests cover the device-change skip, the mid-injection abort, the drain, and the drain assertion in the retarget helper.

### `HARD19-018` — Disk-autostart injection bypassed the shared keyboard-buffer FIFO

**Issue:** The kernal-fallback injector had been the queue for remote-input keystrokes, but disk autostart (`injectDiskAutostart`, `loadFirstDiskPrgViaDma`) issued keyboard-buffer writes outside that single FIFO. Interleaving disk-autostart injection with remote-input injection through separate paths risked ordering hazards and duplicated the queue logic, and the injector's name (`enqueueKernalFallbackInjection`) no longer described its broader role. All keyboard-buffer injection needed to serialize through one queue.

**Fix:** The injector export was renamed to the neutral `enqueueKeyboardBufferInjection` (forwarding `pollIntervalMs`/`maxAttempts`), and `injectDiskAutostart`, `loadFirstDiskPrgViaDma`, and the four remote-input dispatchers were all routed through the single FIFO. This unifies keyboard-buffer injection ordering and lets the device-affinity/drain machinery from HARD19-017 cover disk autostart too. Tests assert the disk-first-PRG and playbackRouter routing go through the shared queue.

### `HARD19-019` — Reset HVSC mid-hydration lets a zombie hydrator resurrect the deleted index

**Issue:** HVSC metadata hydration runs for minutes on a real ~60k-song library and held the pre-reset snapshot object with no cancellation input. "Reset HVSC" only guarded against active ingestion, so a reset mid-hydration proceeded — deleting the library and browse index — while the zombie hydrator kept issuing tens of thousands of failing native reads (a Diagnostics flood and the HVSC-starvation load), stamped every survivor `metadataStatus: "error"` (permanently excluding them from future hydration), and re-persisted the deleted browse index every 5 seconds and on the final chunk, resurrecting what the reset had cleared. Worst case, an immediate reinstall's fresh snapshot was overwritten by the zombie's stale all-error snapshot.

**Fix:** A new module-level hydration generation was added (`hvscHydrationControl.ts`); `resetHvscLibraryData` and ingestion start (`installOrUpdateHvsc`/`ingestCachedHvsc`) bump it, and `ensureHvscMetadataHydration` captures the generation at run start and passes a `shouldContinue()` per-chunk check plus a persist guard bound to it. When the generation changes, `hydrateHvscMetadata` bails at the next chunk boundary (no more reads, no `"error"` stamping) and every `onSnapshotUpdated` persist and the final `setBrowseSnapshot` are skipped, so a stale run can neither resurrect the index nor clobber a reinstall. Tests cover the hydrator chunk-boundary abort (no survivor errors) and a service-level reset mid-hydration that stops reads at 8 and never persists.

### `HARD19-020` — Diagnostics "Clear all" leaves the pinned health-check result

**Issue:** Diagnostics "Clear all" wiped every evidence store — logs, traces, activity, latency samples, health history, recovery evidence — and confirmed with a toast, but did not clear `healthCheckState.latestResult`. Since `useHealthState` derives the global badge from `latestResult` first (per HARD19-004 with no staleness bound), the badge kept asserting the pre-clear verdict indefinitely, now over an empty health history and with no supporting trace evidence. The user's explicit reset gesture failed on the one signal they most wanted reset, and a red badge over an empty evidence set read as "the app is broken".

**Fix:** `handleClearAll` now also calls `setHealthCheckStateSnapshot({ latestResult: null })`, so the clear removes the pinned verdict along with the rest of the diagnostic state. This lets the live trace-derived rollup take over immediately after a clear. The prior bug-encoding test was flipped to assert `latestResult` is null after clear-all.

### `HARD19-021` — playItem bakes the fallback duration without the "default" marker

**Issue:** When `playItem` resolves no songlength (mid-install, after a load failure, or during the HARD19-016 window), it writes `durationMs: resolvedDuration` (the default 3:00 fallback) into the playlist item — but never set `durationSource: "default"`. From that moment the fallback was indistinguishable from a genuinely resolved duration: `applySonglengthsToItems` skipped the item (its gate excludes items whose `durationMs` is set and `durationSource !== "default"`), so once songlengths later became available the item was never re-resolved. Every long song in that batch was cut off by auto-advance at exactly 3:00 forever, recoverable only by removing and re-adding it.

**Fix:** The end-of-launch bake now stamps `durationSource: "default"` when the written duration came from the fallback (`resolvedDurationBase === undefined`), so `applySonglengthsToItems` re-resolves it once songlengths load — the marker HARD9-008 introduced for exactly this purpose. A genuinely resolved duration, and a user's manual "Default duration" slider override, are left untouched. Tests assert the fallback bake tags `"default"` and stays re-resolvable, with a contrast case proving a genuine songlength is not marked.

### `HARD19-022` — Disk autoplay silently reconfigures the physical drive mode

**Issue:** `ensureDiskAutoplayDriveReady` compared the polled drive `type` against an extension-mapped mode and issued `setDriveMode` on any mismatch — even a 1571 reading a D64, which it reads natively, so the change was unnecessary and silently destroyed the user's deliberate drive configuration with no toast and no restore. A mixed D64/D71 playlist thrashed the physical drive mode on every auto-advance, and a secondary defect read the stale pre-`driveOn` snapshot for the type check, plus there was no drives-query invalidation so the drive cards showed a stale type. It was a cross-page trust failure the user only discovered later when 1571-specific software broke.

**Fix:** A compatibility matrix (d64 accepts 1541 or 1571; d71 needs 1571; d81 needs 1581) means the code only switches when the current mode cannot read the image, preserving a compatible mode; unknown/missing type falls through to the safe canonical default. It re-reads drive info after `driveOn` so the decision uses the real type, surfaces a toast on any genuine switch, and (per decision D3) invalidates the `["c64-drives"]` query via the shared registry after any drive mutation so the cards refresh. Tests cover mode preservation, the required-switch-plus-toast, the post-`driveOn` real-type decision, and the query invalidation.

### `HARD19-023` — fetchAllConfig silently accepts partial snapshots

**Issue:** `fetchAllConfig` tolerated per-category read failures by omitting the failed categories, with no signal to the caller. So Save-to-App could persist an incomplete config profile, and the revert baseline could permanently miss categories that failed to read, all while the app reported success. A user who saved or set a baseline during a transient read failure silently under-delivered, with no indication anything was missing.

**Fix:** `fetchAllConfig` now returns `{ configs, failedCategories }`, and the config snapshot carries a `failedCategories` list that marks it provisional: a provisional baseline no longer short-circuits capture (it is re-attempted on later idle windows until complete), `saveCurrentConfig` surfaces `failedCategories` so Save-to-App warns with a destructive toast, and revert flags `baselineIncomplete`. This makes snapshot incompleteness visible instead of silently accepted, per decision D4. Tests cover the provisional-baseline re-capture and the save-side `failedCategories` surfacing.

### `HARD19-024` — Revert verification counts an unreadable category as N false mismatches

**Issue:** On revert, the post-apply verification re-read all categories via the same partial-tolerant `fetchAllConfig`. If a category read failed (plausible while the device is still settling from the batch write it just absorbed), `collectConfigRevertMismatches` mapped the omission to empty actual items, so every item in that category counted as a mismatch — the user was told "N settings did not match" for a revert that actually fully succeeded. The mismatch list showed every item with a null actual value, indistinguishable from a genuine write failure, inverting the trust impact of HARD19-023.

**Fix:** `collectConfigRevertMismatches` now skips unread categories, classifying them as `unverifiedCategories` rather than mismatches, and revert returns them so HomePage words the result "reverted — with caveats" instead of inventing false mismatches. This was implemented together with HARD19-023 since both center on `fetchAllConfig`'s `failedCategories`. A test asserts an unreadable category on revert is classified unverified and does not appear as mismatches.

### `HARD19-025` — Flash save/load/reset falsely toast success on firmware rejection

**Issue:** `saveConfig`, `loadConfig`, and `resetConfig` in the C64 API did not assert that the firmware accepted the operation, so a firmware-rejected flash save/load or reset-to-default could return without throwing and the UI would falsely toast success. The firmware can report an in-body error while the HTTP status looks fine, and these paths swallowed it. A user could believe their configuration was flashed or reset when it was not.

**Fix:** `saveConfig`/`loadConfig`/`resetConfig` now call `assertConfigWriteAccepted(response, {category})` with `flash-save`/`flash-load`/`reset-defaults`, so a firmware rejection throws instead of falsely reporting success. This routes flash operations through the same acceptance-assertion used elsewhere for config writes. A `c64api.branches` test asserts these paths reject on firmware errors.

### `HARD19-026` — useHvscLibrary runs its full lifecycle regardless of hvsc_enabled

**Issue:** `PlayFilesPage` calls `useHvscLibrary()` unconditionally and only the HVSC UI was gated by the `hvsc_enabled` flag; the hook's own effects (`refreshHvscStatus`, `recoverStaleIngestionState`, and — once installed and ready — `ensureHvscMetadataHydration`) checked only native-bridge presence, never the flag. So a user who installed HVSC then disabled it (a supported flow, and default-off but user-visible on the C64U Remote variant) still ran a native status call and minutes of `getHvscSong` hydration on every Play visit, recreating the HVSC-starvation load that knocks Remote Input into a false "Reconnecting". On C64U Remote, Remote Input is the app.

**Fix:** `useHvscLibrary` now takes the resolved `hvsc_enabled` flag and its three lifecycle effects early-return when disabled, with the flag added to their dependency arrays so re-enabling resumes normally; `PlayFilesPage` passes `shouldShowHvscControls(featureFlags)`. The flag was passed as a hook parameter (dependency injection) rather than read via `useFeatureFlags` inside the hook, because `useFeatureFlags` throws without a provider and the 70 existing `renderHook` sites lack one; the CI hook-contract test was updated for the new call shape and the existing sites pass `true`. Tests assert the hook is inert (no status/recover/hydration) when disabled despite an installed bridge, and resumes on an enable flip.

### `HARD19-027` — Health rollup treats an unreachable optional service as whole-device Unhealthy

**Issue:** The Diagnostics health check probed FTP and Telnet unconditionally whenever REST succeeded, with no capability/enablement gate, and `rollUpHealth` returned overall Unhealthy if any single contributor was Unhealthy. On a device with the FTP server off or the Telnet CLI disabled (a common/default c64u configuration), the probe's connection-refused/timeout made that contributor Unhealthy and dominated the rollup, so a device that was perfectly functional for playback, config, disks, and remote input showed a red "Unhealthy" verdict — which then pinned the global badge via HARD19-004. This composed badly with HARD19-004, since one false Unhealthy verdict from an optional service could then pin the global badge red indefinitely.

**Fix:** Per decision D5 the rollup was reweighted rather than the probe gated: an FTP/TELNET contributor failure with REST Healthy now caps the overall result at Degraded, reserving Unhealthy for the REST/control plane. Defensive optional chaining was added for partial-contributor callers. Tests add rollup cases (REST ok plus FTP/Telnet failed → Degraded) and update the Telnet-alone case.

### `HARD19-028` — Discovery interstitial cannot be acknowledged

**Issue:** The device-discovery interstitial's gate did not track acknowledgement, so dismissing it ("Not now" / "Open Settings") did not stop it from re-gating, and the background rediscovery schedule did not re-arm cleanly on the gate edge. A user who dismissed the interstitial at startup (device off) had no clean path to auto-reconnect once the device came on without further manual action. This blocked the intended cold-start recovery flow (a P1 recovery concern).

**Fix:** An `acknowledged` flag and `acknowledgeDeviceDiscoveryResults()` were added to the discovery store (cleared on a new scan); dismissing via "Not now"/"Open Settings" acknowledges, the gate excludes acknowledged results, and `ConnectionController` re-arms the background schedule on the gate's true→false edge with a value-equality bail. This makes the two-device cold-start recovery work: startup with the device off, dismiss the picker, power the device on, and it auto-reconnects. Tests cover the resume-after-ack, no-bump-on-unrelated-emit, the interstitial acknowledgements, and the discovery-manager ack unit behaviour.

### `HARD19-029` — Settings export/import drops two Device Safety values

**Issue:** `restMaxConcurrency` and `machineInputCooldownMs` are live, user-editable Device Safety rows consumed at runtime (the former serializes native REST to protect wedge-prone firmware; the latter throttles machine input), yet `exportSettingsSnapshot` omitted both and `importSettingsJson` never wrote them. A user who tuned REST concurrency down to keep a fragile c64u stable silently got default concurrency back on every transfer, under a "Settings imported" success toast, and a hand-edited file carrying `machineInputCooldownMs` was rejected wholesale as an unknown key. The export/import migration path quietly undid the user's own device-load protection.

**Fix:** The export now includes both fields (added to the payload type), and import applies them via the existing clamping save functions, falling back to defaults for pre-19 files; both are kept in `DEVICE_SAFETY_OPTIONAL_IMPORT_KEYS` so old files still import. A stale legacy test that asserted the export deliberately omits `restMaxConcurrency` was flipped under the test litmus — the setting was never removed from the app, only dropped from transfer, so a user-editable device-safety value must round-trip. Tests cover the round-trip with a non-default sentinel, the optional-key import, and that a legacy import applies the value.

### `HARD19-030` — Imported screen orientation not applied or reflected until relaunch

**Issue:** `importSettingsJson` persisted and broadcast `screenOrientationMode`, but nothing on the import path applied it to the device or refreshed the Settings selector — only the manual `commitScreenOrientationMode` and the startup hook call `applyScreenOrientationMode`. So importing a file with a different orientation left the selector showing the pre-import value (inviting the user to "fix" it and clobber the import) while the device rotated only on the next launch — the exact surprise-rotation the orientation commit path exists to prevent. Any migration where the exported orientation differed hit this.

**Fix:** `handleImportSettings` now mirrors `commitScreenOrientationMode`: it calls `setScreenOrientationMode` and `applyScreenOrientationMode` with the freshly loaded value, so an imported orientation applies to the device and refreshes the selector immediately. A Settings test imports a landscape value and asserts `applyScreenOrientationMode("landscape")` is called and the Landscape selector button becomes active.

### `HARD19-031` — Home Reset and Power Off never publish the machine takeover

**Issue:** The HARD18-022 takeover contract — publish a machine-takeover event on a machine-interrupting action so Play stops auto-advancing over it — was missed by the two most basic actions, Home Reset and Power Off. So a playlist armed with auto-advance kept its timeline running after the user reset the C64 to BASIC (or powered the U64 family off), and at the nominal track end it launched the next track over the freshly reset machine or fired launches at a dead device, producing error toasts and failure evidence against a device the user deliberately interrupted. The two simplest machine-interrupting actions had no takeover publish.

**Fix:** Reset and Power Off now publish the takeover, success-gated (only on a successful action, matching the three already-fixed handlers), via the shared `publishMachineInterrupt` helper. This brings them in line with Reboot/Power Cycle's shipped semantics. Tests cover the power-off takeover with success-gating and the MachineControls Reset publish.

### `HARD19-032` — Reset-family paths don't restore a pending pause-mute

**Issue:** When the machine is paused (with the SID mixer muted by the pause-mute bracket) and the user then triggers a reset-family action, the reset marks the machine running again without first restoring the muted mixer, leaving the SID volumes stuck muted after the reset. The reset paths had no pause-mute bookkeeping, so a reset-while-paused silenced audio until manually corrected. This is the mixer-restore counterpart to the pause-mute work.

**Fix:** The reset-family success paths now restore the pending pause-mute before marking the machine running, going through `publishMachineInterrupt` and using a single `updateConfigBatch` write (per the settled config-transport rule, best-effort so it never blocks or fails the reset UX). This ensures a reset-while-paused leaves the mixer in the correct un-muted state. Tests cover the `machineInterrupt` reset-while-paused restore and a reboot-while-paused mute restore.

### `HARD19-033` — Crashed page offers no retry (SwipeNavigationLayer's boundary copy diverged)

**Issue:** The error boundary used inside `SwipeNavigationLayer` was a local copy that lacked the retry and `active` support of the boundary defined in `App.tsx`, so a page that threw inside the swipe layer showed a dead fallback with no "Try again". The duplicated boundary meant recovery behaviour depended on which boundary caught the error, and the two had drifted. A user hitting a page crash in the swipe layer had no in-app recovery path.

**Fix:** A single `PageErrorBoundary` (with retry and `active` support) was consolidated, and the SwipeNavigationLayer copy was deleted; a new i18n key `app.error.tryAgain` backs the button. Per decision D8 the plan said to import from `App.tsx`, but it was instead extracted to its own module (`src/components/PageErrorBoundary.tsx`) and re-exported from `App.tsx` to avoid an App↔SwipeNav import cycle, with identical behaviour. A SwipeNavigationLayer test exercises the shared fallback and the "Try again" recovery.

### `HARD19-034` — Saved-device background health maintenance never runs

**Issue:** The saved-device health-check hook's `backgroundMaintenance` mode (selected-device-only, freshness-gated, circuit-open-deferred, visibility-resume) was dead code because its `enabled` flag was false in exactly that mode — with the device picker closed, no background health probing occurred even with more than one saved device. This diverged from the F-DIAG-1 design, which documents that background maintenance should run, and meant the freshness/circuit gates engineered for it never fired. The mode existed but was never activated.

**Fix:** Per decision D6, background maintenance was enabled (`enabled = canSwitchDevices`, context split unchanged), so with more than one device saved and the picker closed it drives the `backgroundMaintenance` mode; the existing freshness and circuit gates bound the added probe traffic. ### `HARD19-035` — C64U Remote exports settings under the other product's filename

**Issue:** `handleExportSettings` hardcoded the download filename to `c64commander-settings.json`, so the C64U Remote build emitted the other product's name, and the settings payload carried no variant marker at all. A user running both side-by-side variants (same repo, same device) ended up with indistinguishable files in Downloads and could silently import the wrong one, bleeding one variant's device-safety tuning and feature flags into the other. Tests cover the filename derivation, the variantId round-trip, the cross-variant warning, the no-variantId-silent path, and the SettingsPage warning toast.

### `HARD19-036` — Saving an unreachable hostname can freeze the Save button ~20s

**Issue:** The native `DeviceDiscovery` plugin serializes every scan on one executor thread, and the silent hostname→IP rescue scan intentionally bypasses the JS single-flight. So saving an unreachable hostname while a "Discover devices" LAN scan was already running queued the rescue behind it with no queue-wait bound (the native timeout only starts once the queued job runs), leaving the Save button busy for roughly the probe timeout plus a 10-second queue plus the rescue scan — twenty-plus seconds of unexplained freeze with no cancel. It also briefly doubled the LAN probe burst.

**Fix:** Per decision D9 (JS-side bound only), `evaluateNewDeviceReachability` now races the rescue `discover()` against a 6-second timeout (`raceRescueScan`, timer always cleared in a `finally`); on timeout Save degrades to a plain "unreachable" verdict (just no IP suggestion) and logs a warning, instead of queueing indefinitely. The optional in-flight-scan reuse and the Kotlin executor change were left out of scope per D9. ### `HARD19-037` — "Persistent" error toasts silently auto-dismiss after the notice duration

**Issue:** The error subsystem's contract (ERROR_POLICY §4) says destructive/S3 toasts stay until dismissed so the user reliably sees failures and can use the toast's Retry action, and a redundant JS auto-dismiss timer deliberately exempts destructive toasts. But the Radix `ToastProvider` duration (the user's Notifications setting, default 4s) applies to every toast root that does not override it, and nothing in the app passed a per-toast duration for destructive variants — so error toasts vanished after ~4 seconds. A user who glanced away missed the failure entirely and reasonably assumed success, on exactly the operations (device control, config writes, file persistence) where a missed error costs real state.

**Fix:** Per decision D7, destructive toast roots now receive `duration={Infinity}` (a per-root override) while notices keep the user-configured provider duration. The `TOAST_LIMIT=2`, dedup, and stale-clear-on-recovery mechanisms still bound the number of live error toasts, so persistent errors cannot stack unbounded. A fake-timer Toaster test asserts destructive toasts persist while notices inherit the provider duration, across four cases.

### `HARD20-001` — Analog stick / SwipePad no pointer-id discipline: a second finger or stray pointer-up kills an active drag

**Issue:** `VirtualJoystick`'s stick zone and `SwipePad` had no pointer-id guard: `handlePointerDown` unconditionally re-captured to any new pointer (overwriting `activePointerIdRef` and resetting the drag origin), and `releaseStick`/`release` were bound to `onPointerUp`/`onPointerCancel` without checking `event.pointerId`, so *any* pointer's up/cancel released the active drag. A stray second-finger graze during two-thumb no-look gameplay (stick + fire button) would either kill an in-progress drag or re-base its origin, sending a wrong direction — production-reachable on every multi-touch phone session.

**Fix:** Commit `6f9a0e38` added the pointer-id guard both components were missing: `handlePointerDown` now ignores a new pointer while one is already active, and `releaseStick`/`release` return early unless the event's `pointerId` matches the active one — while still honoring `onPointerCancel` for the active pointer (safety-critical: a genuine cancel of the tracked touch must still release). Tests in `VirtualJoystick.test.tsx` and `SwipePad.test.tsx` cover the `pointerdown(1)→move(1)→pointerdown(2)→pointerup(2)` sequence (held input survives, pointer 1's moves still apply) and confirm `pointercancel` on the active pointer still releases.

### `HARD20-002` — Keyboard held-set lacks shared-input contribution tracking: an atomic shifted tap (F2/F4/F6/F8, CLR, INS) releases SHIFT out from under a hold or SHIFT LOCK

**Issue:** `useKeyboardHoldDispatch`'s `removeFromHeld` deleted a named input from the shared held set unconditionally, with no notion of which key contributed it or whether another holder (SHIFT LOCK, a physically held modifier, another simultaneously-held chord) still needed it. Six production keys (F2/F4/F6/F8/CLR/INS) carry `left_shift` in their chord per `specialKeyMapping.ts`, so tapping any one of them while SHIFT LOCK was on deleted `left_shift` from the held set; the HARD18-002 shift-lock watchdog then saw the held set lacking `left_shift` and silently turned SHIFT LOCK off. The same clobber broke a physically-held SHIFT (UI stayed latched, device typed unshifted) and corrupted overlapping held chords (e.g. INS+F2, releasing INS also dropped F2's shift).

**Fix:** Commit `6f9a0e38` added per-input contribution counting mirroring the HARD19-003 D-Pad fix: `addToHeld` increments a count per contributing input, `removeFromHeld` only clears an input from the shared set when its count reaches zero, SHIFT LOCK holds its own persistent reference on `left_shift`, and a physically held modifier holds a separate reference via `physicallyHeldRef`. Counting is applied at the held-set level, before the diff-to-batch step, so a bare tap (press+release in one flush) still collapses to the firmware `tap` event via `collapseTransientKeyboardTaps`, and the HARD18-002 watchdog is unaffected (it now never observes a false clear from an atomic shifted tap). Tests in `useKeyboardHoldDispatch.test.ts`, `TypeKeyboard.test.tsx`, and `QuickKeysBar.test.tsx` cover SHIFT LOCK + F2 tap, physical-SHIFT + F2 tap, overlapping held chords (INS+F2), and a plain no-shift tap leaving the held set empty (no leaked count).

### `HARD20-003` — FTP writes are in-flight-coalesced by path with no payload identity: a concurrent second write of different content is silently dropped and reported success

**Issue:** `withFtpInteraction`'s in-flight dedup keyed on `host:port|operation:path` and returned the *first* call's promise for any later call with the same key — correct for idempotent `list`/`read`/`ping`, but wrong for `write`: a second `writeFtpFile` to the same path starting while the first was still in flight received the first write's promise, so its bytes were never sent while the caller's success path (toasts, write-back bookkeeping) still ran. The disk write-back subsystem writes full multi-second `.d64` transfers on eject, so a rapid re-mount→save→eject cycle or overlapping config/REU write to the same `/Temp` target could silently discard the newer content while reporting success.

**Fix:** Commit `d28e4cc8` excludes non-idempotent operations from in-flight coalescing: only `list`/`list-recursive`/`read`/`ping` consult/populate `ftpInflight`; `write` (and any future non-idempotent op) always schedules its own task, serialized by the existing scheduler rather than deduped. Test in `deviceInteractionManager.test.ts` blocks a first write to a path, starts a second write to the same path with different payload, and asserts both handlers run (the second dispatching only after the first settles) while same-path reads still coalesce.

### `HARD20-004` — Stop/reset/pause don't cancel a pending coalesced Next/Previous: the 120ms skip timer fires afterwards and relaunches playback

**Issue:** Next/Previous taps coalesce through `pendingUserSkipRef` with a 120ms re-arming timer, but nothing that stops playback (`handleStop`, the machine-takeover subscription, `handlePauseResume`'s pause path) cleared it. Because `playItem` always claims a fresh generation on entry, a still-armed skip firing after Stop passed the HARD18-009 supersede check and relaunched a track on a machine the user had just stopped or reset — including a full reboot+mount+autostart burst for disk items landing right after a reset.

**Fix:** Commit `f9e87329` added a `cancelPendingUserSkip()` helper (clears the timer, resolves all queued resolvers so no awaiting caller hangs, nulls the ref) called from `handleStop`, the machine-takeover handler, and `handlePauseResume`'s pause path, plus a defense-in-depth gate on `flushPendingUserSkip` requiring `isPlayingRef.current || isPausedRef.current` (mirroring the existing unmount-flush guard). Fake-timer tests in `usePlaybackController.test.tsx` cover Next→Stop-within-120ms (playItem not re-invoked, `isPlaying` stays false) and the mirrored machine-takeover case. HIL-verified on C64U: Next then Stop within ~100ms leaves the machine stopped, no relaunch.

### `HARD20-005` — Remounting the SAME materialized disk overwrites the device work file with pristine source bytes, destroying in-game saves

**Issue:** `dropOrFinalizeStaleMaterializedMount` early-returned for a same-id remount (`stale.disk.id === nextDisk.id`) without finalizing, treating it as "nothing stale" — but the work file at that moment could hold newer bytes (in-game saves) than the source. `tryMaterializeDiskMount` then re-resolved and uploaded the pristine source-derived blob, silently discarding every save written since the first mount; the eventual eject even reported success while persisting the rolled-back image. Reachable via any disk replay from Play or a repeat mount of the same library row without an intervening eject.

**Fix:** Commit `aa38fccf` changed the same-id branch to `await readBackAndPersist(stale, ftp)` (finalizing the work file's current bytes to source) before returning, so the subsequent re-materialization uploads the just-saved image; a finalize failure logs via `addErrorLog` and never blocks the remount, matching `finalizeDiskWriteBack`'s never-throw contract. Device-mismatch same-id and different-disk paths are unchanged. Test in `diskMount.test.ts` mutates a fake FTP dep's work-file bytes and asserts a same-id remount finalizes the mutated bytes rather than overwriting with source. HIL-verified on C64U: wrote a save via BASIC to a mounted writable D64, triggered a same-id remount (library row → same drive, no eject) through the app UI, and confirmed the save survived; also verified through a subsequent eject.

### `HARD20-006` — Native Telnet read maps remote-close (EOF) to "no data"; `isConnected` never reflects a peer close

**Issue:** `readPayloadWithinBudget` folded `read() == -1` (EOF/peer closed) into the same `firstRead <= 0` branch as a genuine 0-byte timeout, returning an empty payload either way, and `isConnected` (`socket.isConnected && !socket.isClosed`) never changes on a *remote* close — only a local `close()` does. A device reboot or Wi-Fi drop mid-session left `ensureConnected`'s existing reconnect loop unreachable: the app kept believing the session was live, `readScreen` returned an empty parsed screen instead of surfacing an error, and sends could silently "succeed" into a dead socket.

**Fix:** Commit `d28e4cc8` (native Robolectric coverage added in `b0ce25ca`) distinguishes `-1` (true EOF) in `readPayloadWithinBudget`: the connection is marked dead and the read rejects with a distinct "connection closed" error, while a genuine 0-byte timeout still resolves empty. JS maps the rejection to a `TelnetError` with a `CONNECTION_CLOSED` code; `telnetSession` treats it as disconnected (clears `authenticated`) so the existing `ensureConnected` retry loop — previously unreachable for this case — runs. Tests: `telnetClient.test.ts` (JS, closed-transport → reconnect instead of empty screen) and `TelnetSocketPluginTest.kt` (native Robolectric, immediate EOF / mid-stream EOF / `isConnected`-false-after-EOF, 18 cases).

### `HARD20-007` — BackgroundExecutionService stop→start race swallows the start, leaving playback with no wake lock or watchdog

**Issue:** Companion `stop()` returned as soon as `stopService()` was requested, but `isRunning` only flips false later in the async `onDestroy`. A `start()` issued in that window saw `isRunning == true` (no symmetric pending-stop awareness, unlike the start direction's `startPendingGeneration`) and returned without starting anything; `onDestroy` then completed and released the wake lock. The JS refcount believed background execution was active and never re-issued a start, so a screen-off session silently lost its keep-alive.

**Fix:** Commit `d28e4cc8` (native Robolectric coverage added in `b0ce25ca`) makes companion `start()` always send the start intent with a fresh generation rather than short-circuiting on `isRunning`, relying on `onStartCommand`'s existing generation-staleness check to make redundant/late intents safe (mirroring the already-existing `startPendingGeneration` pattern for the stop direction). Tests: `backgroundExecutionContracts.test.ts` (JS, start-always-sends-intent) and `BackgroundExecutionServiceTest.kt` (native Robolectric, `companionStartSendsFreshIntentEvenWhenAlreadyRunning` and `startRacingStopDestructionRestoresRunningServiceWithWakeLock`, 28 cases total).

### `HARD20-008` — Cancelling a native HVSC download doesn't stop the native transfer; a prompt retry races the orphaned download on the same cache path

**Issue:** The native download branch called `Filesystem.downloadFile(...)` with no abort/signal mechanism, and cancellation only threw at the *next* `ensureNotCancelled()` check — reached only after the native transfer resolved. Cancelling a multi-hundred-MB HVSC install left the native HTTP transfer running to completion in the background (minutes of bandwidth/battery after the UI reported "Cancelled"), and an immediate retry started a second `Filesystem.downloadFile` to the same cache path while the orphaned first transfer might still be writing it, risking interleaved writes that a size-only `*0.99` check could miss.

**Fix:** Commit `eceef117` implements the review's low-risk option: on cancel, the archive path is recorded as a cancelled-orphan; a retry to the same path waits for the orphan to settle (stable size across a short interval, successful `deleteCachedArchive`) before starting a new native download, and final size/checksum validation against the marker is now a hard gate (not just a recorded value) when `expectedSizeBytes` is known. Test in `hvscDownload.test.ts` uses a fake `Filesystem` whose `downloadFile` hangs: cancel → retry asserts the second download does not start until the first settles.

### `HARD20-009` — Home "Menu" doesn't pause Play's timeline or block auto-advance: the next track can launch over the open Ultimate menu

**Issue:** `handleMenuToggle` only called `deviceControl.toggleMenu()`, unlike every other session-interrupting Home action (Reset, Reboot, Power Off/Cycle all publish a machine interrupt per HARD19-031). Opening the Ultimate menu suspends the running C64 session at the hardware level, but Play's persisted timeline kept counting against the frozen machine; returning to Play (or the guard coming due while still mounted) could fire a full `handleNext("auto")` launch — reboot/mount/autostart for disk items — yanking the machine out of the menu the user was actively navigating.

**Fix:** Commit `f9e87329` publishes a pause-like suspend on a successful menu open (clearing `autoAdvanceDueAtMs`/the native due-time, mirroring the existing external-pause handling from HARD19-009) and resumes/re-arms from remaining time on close; because the client-side `menuOpen` estimate is explicitly non-authoritative, the armed state also clears on any subsequent user transport action, machine interrupt, or device switch so a drifted estimate can never strand playback paused. Test in `HomePage.test.tsx` covers the open/close state mirroring. HIL-verified on C64U: opened the Ultimate menu from Home during a 33s SID track and held it open 40s (past track end) — the menu was untouched throughout; on close, Play's timeline showed the clock had frozen at ~1s elapsed the entire time and resumed correctly.

### `HARD20-010` — Auto-advance dies while the user browses other tabs: the native watchdog fires into a removed listener; playlists stall at track end with the screen on — MITIGATED

**Issue:** `PlayFilesPage` unmounts whenever another tab is active (`SwipeNavigationLayer` renders inactive slots as placeholders), which removes the `backgroundAutoSkipDue` listener along with the page's JS timeline interval. The BUG-040-hardened unmount path correctly keeps the foreground service and native due-time watchdog armed across unmount, but when the watchdog fired with no listener attached, `notifyListeners` dropped the event and the one-shot native due-time cleared itself with no retry — so a playing SID/MOD track kept sounding indefinitely past its songlength until the user returned to Play and session restore lurched the stale guard forward.

**Fix (mitigation, not full fix):** Commit `f9e87329` adds `retainUntilConsumed: true` to the native `notifyListeners("backgroundAutoSkipDue", ...)` call, so an event fired while unmounted is delivered on remount instead of dropped — converting "stalls until the user returns AND the boundary is re-derived" into "advances immediately on return." Test in `backgroundExecutionContracts.test.ts` asserts the retain flag. HIL-verified on C64U: played a 33s track, navigated to Home for 35s (tab-nav, page unmounted) with the screen on, then returned to Play — playback was actively continuing (elapsed 0:11 into a fresh playthrough), no stall.

### `BUG-025` — Background-execution wake lock leaks after Stop once the Play page has been remounted while playing

**Issue:** After the Play page was unmounted and remounted while playback was active (e.g. navigating Play → Home → Play), pressing Stop returned the UI to idle but never released the Android wake lock, so `dumpsys power` still showed `Wake Locks: size=1 c64commander:background_execution` and the `BackgroundExecutionService` kept running many minutes later, persisting until the app process was killed. ### `BUG-027` — Android hardware Back navigates the route instead of dismissing an open non-modal overlay

**Issue:** Pressing the Android hardware Back button to dismiss any open non-modal Radix overlay (dropdown menu, Select, Popover, context menu) instead bounced the user to the previous route, so for example opening a disk-list `⋮` dropdown or a Drive Type Select on `/disks` and pressing Back popped `/disks → /settings` rather than just closing the overlay. ### `BUG-028` — Android Back globally trapped at tab roots after the BUG-027 fix

**Issue:** After the first iteration of the BUG-027 fix, once any page containing a Radix Select or DropdownMenu had been shown (i.e. almost every screen — Home Quick Config, Config rows, Settings, Disks), Android Back at a tab root no longer backgrounded or exited the app and was instead silently consumed as a no-op `Escape`. ### `BUG-034` — Config Drift diagnostics view issues a destructive `PUT /v1/configs:load_from_flash` on open

**Issue:** The read-only-looking "Config Drift" diagnostics panel actually mutated the live device, because `computeConfigDrift()` (`src/lib/diagnostics/configDrift.ts`) called `api.loadConfig()`, which is `PUT /v1/configs:load_from_flash` — an operation that overwrites the device's RUNNING config with the flash-saved values, silently discarding any unsaved runtime config changes, and although a code comment claimed a save-snapshot/restore-runtime approximation, no snapshot-save and no restore were implemented so runtime was left equal to flash permanently. A secondary defect was that `computeConfigDrift` fired an unpaced burst of ~2N+1 sequential REST calls that tripped the c64u overload behaviour into "Connection reset", and a tertiary defect leaked that raw JSON error into the Problems log. This was High severity: a diagnostics view must not mutate device state per the C64U safety policy, and on-device evidence (iter62) captured the `rest.put /v1/configs:load_from_flash` in the app's Activity log while host-side curl proved the device was fully responsive, showing the destructive PUT was genuinely dispatched.

**Fix:** The destructive `await api.loadConfig()` and the post-load persisted re-fetch were removed from `configDrift.ts`, making Config Drift strictly read-only so it no longer mutates the device or fires the double request burst; because the firmware exposes no non-destructive persisted-config endpoint, the view now returns a clear informational error explaining the limitation instead of silently loading flash, and the unit test `tests/unit/lib/diagnostics/configDrift.test.ts` was updated to assert NO device API calls. It was FIXED and HIL-validated on c64u (fw 1.1.0): opening Config Drift and tapping in-panel Refresh rendered the read-only message and issued ZERO device requests, with package-filtered logcat showing 0 `load_from_flash`/`Connection reset` lines versus the pre-fix 2× `load_from_flash` plus the reset cascade. It was re-confirmed in iter63 and again in iter66 after being committed at `6258ec5` (zero `/v1/configs` traffic during open plus three refreshes), and marked CLOSED.

### `BUG-040` — Background-execution guard released when leaving the Play tab while playing

### `BUG-064` — CPU Speed slider on Home silently fails for single-digit speeds

### `BUG-072` — Play page filter checkboxes obscured by the bottom TabBar overlay

**Issue:** On the Play route, the SID/MOD/PRG/CRT/Disk filter checkbox rows sat inside the bottom TabBar's hit band, so tapping a visible checkbox routed to Home instead of toggling the filter — on a 1080×2280 Pixel 4 the TabBar was at `y:1980-2087` while the SID/MOD/PRG row was at `y:2013-2059` (fully inside the band) and the CRT/Disk row at `y:2079-2128` (partially under it) — and repeated droidmind and mobile-mcp taps all triggered Play→Home navigation, with no `toggle` action ever recorded.
