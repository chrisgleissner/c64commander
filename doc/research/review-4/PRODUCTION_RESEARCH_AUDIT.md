# PRODUCTION RESEARCH AUDIT - C64 Commander (Review 4)

## 1. Executive Summary

- Audit date: 2026-02-19.
- Scope completed across all planned phases, with runtime-interaction blockers explicitly recorded where device keyguard prevented unlocked UI access.
- Total issues documented: 110 (`C64C-001` to `C64C-110`).
- Highest-risk findings:
  - Extremely high redundant REST activity in golden traces (`78.11%` duplicate requests within 500 ms).
  - Startup jank on device with repeated main-thread frame skips (`Skipped 96-106 frames`).
  - Recurrent startup Filesystem plugin errors (`Directory does not exist`, `File does not exist`).
  - iOS background execution bridge is explicit no-op, creating platform reliability parity risk.
  - HVSC Android JVM test failures (`NoClassDefFoundError` / `ClassReader`), reducing confidence in ingest robustness.

## 2. Environment Description

- Repository: `/home/chris/dev/c64/c64commander`.
- Branch and commit captured in `doc/research/review-4/metrics/environment.txt`.
- Runtime target device (evidence source): Samsung Galaxy Note 3 (`SM-N9005`), Android 11 (SDK 30), serial `211...`.
- ADB inventory showed additional devices (S21 FE and emulator), but runtime evidence used only Note 3 outputs.
- Important constraint: requested serial was `211...`; adb in this environment resolves lowercase `211...`.

## 3. Measurement Methodology

- Build and install validation:
  - `npm run cap:build`
  - `cd android && ./gradlew assembleDebug`
  - `adb -s 211... install -r <apk>`
- Startup timing:
  - cold and warm launch loops with logcat timestamp correlation.
- Runtime signals:
  - focused logcat extraction for startup, lifecycle, plugin errors, and rendering.
- Network behavior:
  - large-scale trace mining from golden Playwright traces.
- Memory and resource profiling:
  - repeated `dumpsys meminfo` and process sampling under cold/warm cycles.
- Static architecture/concurrency audit:
  - targeted source scans of hooks, query invalidation points, bridge code, timers, listeners, retries, and logging.

- Build and deployment succeeded with debug APK `0.1.0` (`versionCode=1772`).
- Dynamic animation reduction feature not found beyond `prefers-reduced-motion` CSS media query.
- Cold/warm startup timings captured.

### Phase 2 - Startup Performance Profiling

- Startup repeatedly shows main-thread stalls and repeated plugin errors.
- Startup warning flood includes hundreds of null-string warnings.
- Capacitor cookie fetch is executed for nearly every intercepted request even though no cookie-set signals were observed in captured startup logs.

## 11. Observed Anti-Patterns

- Broad predicate-based query invalidation used as default refresh strategy.
- Route and visibility events triggering full c64 query invalidations.
- Multiple independent pollers for shared device info state.
- Bridge-level startup work executed before first interaction readiness.
- Warning-heavy logs reducing actionable signal-to-noise ratio.
- Large cross-platform behavior gaps in background execution semantics.

## 12. Systemic Architectural Risks

- Tight coupling between route lifecycle and data-refresh behavior creates request amplification.
- Connection state transitions mix discovery, demo fallback, sticky lock behavior, and cache invalidation effects.
- Legacy and current playback paths coexist, increasing hidden lifecycle side-effects.
- HVSC runtime handles multiple complex modes with singleton mutable state and heavy memory operations.
- Diagnostics and trace channels are rich but not yet tied to strict runtime SLO guardrails.
